1 How the Current System Works

Core Data Structures

The violation system is built on three nested layers of data that live on each NonNegotiable:

Structure Purpose Key fields
LockConfiguration Top-level lock settings (28/60/90-day commitment) totalDays, windowLengthDays = 14, windows: [Window]
Window A 14-day sliding sub-period within the lock weeklyViolationCount: Int, weeksEvaluated: Set<WeekID>
Violation A single recorded violation event date: Date, kind, windowIndex: Int, weekId: WeekID

Violation.kind has two possible values:

  • .missedDailyCompliance — used for daily mode protocols. Recorded once per missed day.
  • .missedWeeklyFrequency — used for session mode protocols. Recorded once per week where the target frequency was not met.

The Two Protocol Modes

Daily Mode

User must complete the protocol every single day. Missing any one day creates a .missedDailyCompliance violation for that day.

Target: 7 completions per week
Violation unit: per day missed

Session Mode

User must complete the protocol N times per week (1–6). Missing the weekly target creates a .missedWeeklyFrequency violation for that week.

Target: 1–6 completions per week
Violation unit: per week missed

The Evaluation Pipeline

The pipeline runs when: (a) app launches, (b) app becomes active from background, (c) the simulated date changes in DevOptions. The entry point is runDailyIntegrityTick().

1
evaluateDailyCompliance

For each daily protocol, checks if yesterday was a missed day (no completion recorded). If missed, appends a .missedDailyCompliance violation and calls updateRecoveryState. Session protocols skip this entirely.

2
evaluateWeekCatchUp

Iterates over fully ended weeks only (weekEnd ≤ startOfToday). For each ended week not yet in weeksEvaluated, checks if the session frequency target was met. If not, appends a .missedWeeklyFrequency violation. The current week is always skipped.

3
evaluateIntraWeekSessionShortfallIfNeeded recently added

New method that checks if a session protocol has mathematically no way to reach its weekly target given remaining feasible days. Only fires when completionCount + remainingFeasibleDays < expected. Blocked by the creation-week grace bug (see Section 3).

4
advanceWindows

Moves each protocol's active window forward if the current 14-day period has elapsed. Creates new window entries.

5
evaluateRecoveryDay

Checks if the required consecutive clean-day count has been met to exit recovery mode. Currently requires 7 consecutive clean days.

2 When Recovery Mode Triggers

The Threshold Rule

After every violation is appended, updateRecoveryState is called. It checks:

func recoveryViolationThreshold(for mode: NonNegotiableMode) -> Int {
    switch mode {
    case .daily:   return 2  // 2 missed days → recovery
    case .session: return 1  // 1 missed week → recovery
    }
}

The count checked is window.weeklyViolationCount — accumulated violations within the current 14-day window. Once the count reaches the threshold, the protocol transitions to .recovery.

ℹ️ These thresholds were recently lowered from daily: 3 → 2 and session: 2 → 1. They are now much more aggressive, which is correct — but the violation recording pipeline itself still prevents violations from being written in the first place (see Section 3).

What Must Happen to Enter Recovery

Mode What triggers recovery Time required Current reality
Daily 2 missed days within the same 14-day window 2+ days of skipping Works in theory, blocked by grace bugs
Session 1 missed week within the same 14-day window Until the week fully ends (Sunday midnight) Never fires during creation week

Recovery Exit Conditions

A protocol exits recovery mode when evaluateRecoveryDay confirms 7 consecutive days without a missed daily or missed weekly violation. During recovery, allocation planning for that protocol is suppressed.

3 Known Bugs — Why Violations Don't Record

🚨 These three bugs combine to make violations never appear for a fresh protocol regardless of how many days the user skips.
🐛 Bug #1 — Creation Week Grace Is Unconditional (Session Mode)
NonNegotiableEngine.swift → evaluateInitialPartialWeeklyGrace()

The grace logic is:

// Current implementation — too broad:
return InitialPartialWeeklyGraceEvaluation(
    shouldSuppressShortfall: nn.createdAt > weekInterval.start,
    // TRUE if created any time after Monday 00:00 — which is ALWAYS true
    creationWeekId: creationWeekId
)

This means: any protocol created after Monday midnight gets full-week grace for the entire creation week. Since users always create protocols during the day (not at midnight), every single session protocol gets an entire free week with zero violations possible.

Impact: evaluateIntraWeekSessionShortfallIfNeeded returns early. evaluateWeekCatchUp also suppresses the creation week. Result: zero violations the entire first week.

🐛 Bug #2 — Session Mode Has No Real-Time Check (Only End-of-Week)
NonNegotiableEngine.swift → evaluateWeekCatchUp()

evaluateWeekCatchUp only processes weeks where weekEndExclusive ≤ startOfToday. The current week is categorically excluded:

// Only processes FULLY ENDED weeks:
guard weekEndExclusive <= startOfToday else { continue }

This means session violations are only ever recorded after a week completely ends (Monday of the next week). If you create a protocol on Monday and skip every day through Saturday, zero violations are written until Monday of the following week — when the system catches up.

Impact: Even if Bug #1 is fixed, violations are delayed by up to 7 days for session mode.

🐛 Bug #3 — Daily Mode Missed a Check on the Creation Day
NonNegotiableEngine.swift → evaluateDailyComplianceIfNeeded()

Daily mode has a creation-day grace period that prevents recording a violation if "yesterday" is the day the protocol was created. This is intentional and correct. However, when the user uses DevOptions to jump days forward, the tick fires for each simulated day but the grace logic only checks the very first day. If the user jumps forward 5 days without completing, violations should accumulate for days 2–5 but only will if each simulated day triggers the tick correctly.

The dev clock triggers runDailyIntegrityTick only once per simulatedNow change — meaning jumping from Day 1 to Day 5 in one DevOptions action fires the tick once, not 4 times. Only Day 5's violation is evaluated; Days 2, 3, and 4 are silently skipped.

Impact: When using DevOptions to skip days, fewer violations are recorded than expected. The system may never reach the recovery threshold even if many days were skipped.

4 Execution Trace — Daily Mode

Following the exact code path for a user who creates a daily protocol and then uses DevOptions to jump 3 days forward:

Day 0 — Protocol Created (Monday 10:00 AM)

Protocol enters .active state. Lock starts. Creation day grace is set.

User jumps to Day 3 (Thursday) via DevOptions in one action

simulatedNow changes once → runDailyIntegrityTick fires once (not once per day).

evaluateDailyComplianceIfNeeded called for Thursday

Checks "yesterday" = Wednesday. No completion on Wednesday → records 1 violation (.missedDailyCompliance). Monday and Tuesday are skipped. weeklyViolationCount = 1.

Threshold check: 1 < 2 → No recovery yet

One violation is not enough. The missed Monday and Tuesday violations were never recorded.

User jumps to Day 4 (Friday) via DevOptions

Another tick fires. Checks Wednesday — already processed? No — checks "yesterday" = Thursday. Records 1 more violation. Count = 2.

Threshold check: 2 ≥ 2 → .recovery entered ✓

Recovery mode triggers. Daily mode does eventually work if the user advances one day at a time.

⚠️ Daily mode only records violations for "yesterday" — never for multiple past days in a single tick. If DevOptions jumps 3 days at once, only 1 violation is added per jump.

5 Execution Trace — Session Mode (Fresh Install)

Following the exact code path for a user who creates a session protocol (e.g., 3×/week) and then uses DevOptions to skip several days:

Day 0 — Protocol Created (Monday 10:00 AM, Week 19)

Protocol active. createdAt = Monday 10:00 AM. Grace evaluation: createdAt > weekInterval.start (Monday 00:00) → TRUE → full-week grace granted.

User jumps to Thursday via DevOptions

runDailyIntegrityTick fires.

evaluateDailyCompliance — skipped (session mode)

Daily compliance check returns early. No violations from this path.

evaluateWeekCatchUp — skipped (week not ended)

Week 19 ends next Monday. Thursday < Monday → week not eligible. Skipped.

evaluateIntraWeekSessionShortfallIfNeeded — blocked by grace

Calls evaluateInitialPartialWeeklyGrace → returns shouldSuppressShortfall = true because createdAt > weekInterval.start. Returns early. Zero violations.

User jumps to next Monday (Week 20 begins)

Now Week 19 is fully ended. evaluateWeekCatchUp processes Week 19 — but checks creation week grace again → suppressed. Still zero violations.

User skips all of Week 20 too

End of Week 20: evaluateWeekCatchUp processes Week 20. No grace for Week 20. Target = 3, completions = 0 → 1 violation recorded. Count = 1 ≥ threshold 1 → recovery triggered.

🚨 A fresh session protocol cannot enter recovery until the second full week ends — meaning at minimum 14+ days of skipping before recovery is possible. This is completely wrong behavior.

6 Redesign Proposal

💡 Core philosophy shift: Instead of accumulating violation counts in window buckets and hoping the grace periods don't suppress them, derive violation status directly from completion records at evaluation time. This makes the system predictable, testable, and impossible to suppress accidentally.

Proposed Architecture: Observation-Based Violation Engine

The redesigned engine works by directly observing completion records and computing per-period compliance in a single deterministic pass — no accumulated counters, no window state, no suppression chains.

Current (window-bucket model)

  • Violations written incrementally as side-effects of ticks
  • Suppressed by grace flags in 5 different places
  • Window weeklyViolationCount is authoritative counter
  • Single-day resolution (only "yesterday" checked per tick)
  • Hard to reason about — bugs compound silently

Proposed (observation model)

  • Compute compliance for ALL past periods on each tick
  • Grace is a single, explicit, named predicate
  • Violation list is the single source of truth (no separate counters)
  • Idempotent: re-evaluating produces the same result
  • Each past period either has a violation entry or doesn't

Redesigned Data Model

Remove weeklyViolationCount: Int from Window entirely. Derive all counts from violations: [Violation] at read time. This eliminates the dual-source-of-truth problem where the list and the counter could disagree.

// Remove from Window:
// var weeklyViolationCount: Int   ← delete this
// var weeksEvaluated: Set<WeekID> ← delete this

// Derive at read time:
extension NonNegotiable {
    func violationCount(in windowIndex: Int) -> Int {
        violations.filter { $0.windowIndex == windowIndex }.count
    }

    var isOverThreshold: Bool {
        guard let wi = activeWindowIndex else { return false }
        return violationCount(in: wi) >= recoveryThreshold
    }
}

Redesigned Evaluation Method

Replace the separate daily/weekly evaluation methods with a single reconcileViolations pass that processes all unresolved past periods in one go:

// Runs once per tick. Processes all past periods not yet in violations list.
func reconcileViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    guard nn.state == .active || nn.state == .recovery else { return }

    switch nn.definition.mode {
    case .daily:
        reconcileDailyViolations(&nn, referenceDate: referenceDate)
    case .session:
        reconcileSessionViolations(&nn, referenceDate: referenceDate)
    }

    updateRecoveryStateFromViolations(&nn)
}

func reconcileDailyViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    let startOfToday = DateRules.startOfDay(referenceDate, calendar: calendar)
    let lockStart    = DateRules.startOfDay(nn.lock.startDate, calendar: calendar)
    let graceCutoff  = calendar.date(byAdding: .day, value: 1, to: lockStart)!

    // Walk every day from lock start up to (but not including) today
    var day = lockStart
    while day < startOfToday {
        defer { day = calendar.date(byAdding: .day, value: 1, to: day)! }

        // Grace: skip the creation day
        if day < graceCutoff { continue }

        let dayId = DayIdentifier(day)
        // Idempotent: skip if already recorded
        if nn.violations.contains(where: { $0.dayId == dayId }) { continue }

        let hadCompletion = nn.completionRecords.contains(where: {
            DayIdentifier($0.date) == dayId
        })
        if !hadCompletion {
            nn.violations.append(Violation(date: day, kind: .missedDailyCompliance))
        }
    }
}

func reconcileSessionViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    let startOfToday   = DateRules.startOfDay(referenceDate, calendar: calendar)
    let lockStart       = DateRules.startOfDay(nn.lock.startDate, calendar: calendar)
    let creationWeekId  = DateRules.weekID(for: lockStart, calendar: calendar)
    let expected        = expectedCompletionsPerWeek(for: nn.definition)

    // Walk every ISO week from lock start up to current week
    var weekStart = DateRules.weekInterval(containing: lockStart, calendar: calendar).start
    while weekStart < startOfToday {
        let weekEnd = calendar.date(byAdding: .day, value: 7, to: weekStart)!
        let weekId  = DateRules.weekID(for: weekStart, calendar: calendar)
        defer { weekStart = weekEnd }

        // Idempotent: already have a violation for this week
        if nn.violations.contains(where: { $0.weekId == weekId }) { continue }

        // Grace: creation week — only suppress if not enough time remained
        if weekId == creationWeekId {
            let feasibleDays = feasibleDaysInWeek(weekStart, weekEnd: weekEnd, createdAt: nn.lock.startDate)
            if feasibleDays < expected { continue } // genuinely couldn't hit target
        }

        let completions = nn.completionRecords.filter {
            $0.date >= weekStart && $0.date < weekEnd
        }.count

        let weekFullyEnded  = weekEnd <= startOfToday
        let impossibleToMeet = completions + feasibleDaysRemaining(from: startOfToday, weekEnd: weekEnd) < expected

        if weekFullyEnded || impossibleToMeet {
            if completions < expected {
                nn.violations.append(Violation(date: weekStart, kind: .missedWeeklyFrequency, weekId: weekId))
            }
        }
    }
}
✅ This approach is idempotent: running the method multiple times produces the same result. Each past period either has a violation entry or doesn't — no duplicate accumulation, no suppression chains.

Redesigned Creation Week Grace

Replace the unconditional grace check with a proportional one. Grace should only be granted if the user genuinely could not have hit the weekly target given when the protocol was created:

// Proposed — proportional grace:
private func feasibleDaysInWeek(
    _ weekStart: Date,
    weekEnd: Date,
    createdAt: Date
) -> Int {
    let effectiveStart = max(weekStart, DateRules.startOfDay(createdAt, calendar: calendar))
    var count = 0
    var d = effectiveStart
    while d < weekEnd {
        count += 1
        d = calendar.date(byAdding: .day, value: 1, to: d)!
    }
    return count
}

// Usage in creation week check:
if weekId == creationWeekId {
    let feasible = feasibleDaysInWeek(weekStart, weekEnd: weekEnd, createdAt: nn.lock.startDate)
    if feasible < expected {
        continue // fair grace — couldn't have hit target anyway
    }
    // Otherwise: user had enough days, evaluate normally
}

Example: User creates a 3×/week protocol on Thursday. Only 3 feasible days remain (Thu, Fri, Sat). Target = 3. Grace applies because 3 ≥ 3 is barely achievable, but if target were 4, grace would NOT apply since 3 < 4 — actually this is wrong: it SHOULD grant grace if feasible < expected, meaning Thursday creation with target 4 gets grace (3 < 4).

Fixing the Multi-Day Skip Problem

The current system only processes "yesterday" per daily tick. The redesigned reconcileDailyViolations walks every day from the lock start to today, so jumping 5 days forward correctly produces 5 violation checks in a single tick (minus the creation day grace).

✅ With the observation model, DevOptions day-skipping works correctly out of the box. No special handling needed.

7 Implementation Plan

Phase 1 — Immediate Fixes (Low Risk)

Fix 1 · Proportional Creation Week Grace

File: NonNegotiableEngine.swift → evaluateInitialPartialWeeklyGrace

Change the suppression condition from nn.createdAt > weekInterval.start to feasibleDays < expected. Unblocks the intra-week session check and the week catch-up check for most real-world scenarios immediately.

Fix 2 · Multi-Day Catch-Up for Daily Mode

File: NonNegotiableEngine.swift → evaluateDailyComplianceIfNeeded

Change the method to walk from the last evaluated day to yesterday, not just check yesterday. Store a lastEvaluatedDay: Date? on the protocol (or derive from the last daily violation date) to know where to resume. Ensures DevOptions day skips produce all missed-day violations.

Phase 2 — Core Redesign (High Impact)

Fix 3 · Replace Window Counter with Derived Count

Files: Window.swift, NonNegotiableEngine.swift, CommitmentSystemStore.swift

Remove weeklyViolationCount and weeksEvaluated from Window. Derive all counts from violations at read time via computed properties. Update all call sites.

Fix 4 · Replace Incremental Evaluation with Reconciliation Pass

Files: NonNegotiableEngine.swift, CommitmentSystemEngine.swift, CommitmentSystemStore.swift

Replace evaluateDailyCompliance, evaluateWeekCatchUp, and evaluateIntraWeekSessionShortfallIfNeeded with a single reconcileViolations method per the design above. Wire it into the tick pipeline.

Migration Considerations

Existing users have weeklyViolationCount stored in their JSON files. A migration is needed:

  • Add a migration sentinel key to UserDefaults (e.g., didRunViolationSystemMigration20260505)
  • On first launch after update: re-run reconcileViolations for all protocols from their lock start date
  • Drop weeklyViolationCount and weeksEvaluated from the Codable init using decodeIfPresent with fallback (they become ignored legacy fields)
  • The violation list itself is preserved — reconciliation is additive and idempotent
ℹ️ The migration follows the same pattern as existing legacy key migrations: decodeIfPresent + fallback in init(from:). No separate migration versioning system is needed.
