1 How the Current System Works

Core Data Structures

The Two Protocol Modes

Daily Mode

Must complete every day. Missing any day creates a .missedDailyCompliance violation for that specific day.

Target: 7/week · Violation unit: per missed day

Session Mode

Must complete N times per week (1–6). Failing the weekly target creates a .missedWeeklyFrequency violation for that week.

Target: 1–6/week · Violation unit: per missed week

The Evaluation Pipeline

Triggered by: app launch, app foregrounding, simulated date change in DevOptions. Entry point: runDailyIntegrityTick() in CommitmentSystemStore.

2 When Recovery Mode Triggers

The Threshold Check

After each violation is appended, updateRecoveryState checks window.weeklyViolationCount against the threshold:

func recoveryViolationThreshold(for mode: NonNegotiableMode) -> Int {
    switch mode {
    case .daily:   return 2  // 2 missed days → recovery
    case .session: return 1  // 1 missed week → recovery
    }
}

The counter checked is window.weeklyViolationCount — accumulated within the current 14-day window. Once it reaches the threshold the protocol transitions to .recovery.

These thresholds were recently lowered (daily 3→2, session 2→1). The thresholds are now correct. The problem is violations are not being written at all — the pipeline that feeds these counters is broken.

Recovery Exit

A protocol exits recovery after 7 consecutive days with no new violations. The clean-day check uses effectiveWeeklyTarget() in CommitmentSystemEngine.swift which itself calls isInitialPartialGraceWeek() — a duplicate of the same grace bug that suppresses violation entry (Bug #3).

3 Confirmed Bugs — Source Verified

🚨 All four bugs below are verified against the current source. Bugs 1, 2, and 4 combine to make session-mode violations impossible to record during the first two weeks of a new protocol. Bug 3 independently corrupts recovery exit timing.
Bug #1 — Creation Week Grace Is Unconditionally Broad (Session Mode)
NonNegotiableEngine.swift:472 → evaluateInitialPartialWeeklyGrace() CommitmentSystemEngine.swift:572 → isInitialPartialGraceWeek() — duplicate of the same bug 
// Current — suppresses ALL shortfalls if createdAt is any time after Mon 00:00:
return InitialPartialWeeklyGraceEvaluation(
    shouldSuppressShortfall: nn.createdAt > weekInterval.start,
    creationWeekId: creationWeekId
)

Key nuance: createdAt is normalized to start-of-day at protocol creation (NonNegotiableEngine.swift:70: createdAt: normalizedStart). A Monday-created protocol gets createdAt = Mon 00:00:00, which equals weekInterval.start, so the condition is false — Monday creation gets no grace and the bug doesn't apply. But Tuesday through Sunday creation always produces createdAt > weekInterval.start = true, granting unconditional full-week grace.

Correct logic: Grace should only apply when the user had fewer feasible days than the weekly target. A user who creates a 2×/week protocol on Thursday (3 feasible days remain) should not get grace — they could still hit the target. A user who creates a 5×/week protocol on Thursday (3 feasible days remain) should get grace because 3 < 5.

Impact: Any session protocol created Tuesday–Sunday gets full first-week immunity from violations. Since nearly all real users create protocols during the day (not at midnight Monday), this affects essentially all session protocols.

Bug #2 — Session Mode Has No Real-Time Check (Week-End Only)
NonNegotiableEngine.swift → evaluateWeekCatchUp() 
// Only processes FULLY ended weeks — current week always excluded:
guard weekEndExclusive <= startOfToday else { continue }

Session violations are only evaluated after a week fully closes. If you create a protocol on Tuesday and skip Wednesday, Thursday, Friday, Saturday, and Sunday — zero violations are written until Monday of the following week. evaluateIntraWeekSessionShortfallIfNeeded was added to fill this gap but is blocked by Bug #1.

Impact: Even after Bug #1 is fixed, session violations are delayed by up to 6 days for in-progress weeks unless the intra-week inevitable-miss check fires first.

Bug #3 — Grace Bug Duplicated in Recovery Exit Logic
CommitmentSystemEngine.swift:535, 572 → effectiveWeeklyTarget() → isInitialPartialGraceWeek() 
// CommitmentSystemEngine.swift:584 — same broad condition:
return nonNegotiable.createdAt > week.start

The clean-day evaluation that governs exiting recovery calls effectiveWeeklyTarget(), which calls isInitialPartialGraceWeek(), which has the same unconditional grace condition. This means:

  • If somehow a session protocol entered recovery during its creation week, the clean-day counter would not count the creation week's days correctly — the effective target is set to 0, making every day look "clean."
  • When Bug #1 is fixed to be proportional, this file must be fixed identically, or recovery exit will still behave incorrectly for creation-week scenarios.

Impact: Recovery exit timing is incorrect for protocols that enter recovery close to their creation date.

Bug #4 — Window Boundary Splits Weekly Completion Count (False Shortfalls)
NonNegotiableEngine.swift:572 → countCompletions() 
return nn.completions.filter { record in
    record.weekId == weekId
    && record.kind == .counted
    && record.date >= weekInterval.start
    && record.date < weekInterval.end
    && record.date >= window.startDate   // ← window boundary filter
    && record.date < window.endDate      // ← window boundary filter
    && record.date >= nn.lock.startDate
    && record.date < lockEnd
}.count

A 14-day lock window runs Monday-to-Sunday-of-week-2 plus Monday-to-Sunday-of-week-3. ISO weeks are also Monday–Sunday. These boundaries don't always align. When the 14-day window boundary falls mid-week (e.g., window ends Thursday), the system evaluates that week under the new window — but completions logged before Thursday are filtered out by record.date >= window.startDate.

Concrete example: Lock starts Monday week 1. Window boundary falls on Monday week 3. A completion logged on Wednesday of week 2 is attributed to window 0. When week 2 is evaluated for session compliance at the end of the week (also window 0), it counts correctly. But if the window rolls over mid-week of a later period, completions from the first half of that week can fall in window N-1 while the shortfall is evaluated in window N.

Impact: Periodic false session violations that report a shortfall even when the user completed the target, every ~14 days at a window boundary crossing. Hard to reproduce because it only triggers at specific week/window alignments.

4 Execution Trace — Daily Mode

✅ Daily mode is largely working. The multi-day catch-up loop (lines 120–132 of NonNegotiableEngine.swift) already handles DevOptions day-skipping correctly by walking lastDailyComplianceCheckedDay + 1 through yesterday. Recovery triggers after 2 missed days.
Protocol Created — Monday 00:00 (normalized)

DevOptions: jump to Wednesday (skip Tuesday)

Tick fires once. evaluationStart = Tuesday (lock start + 1, since Monday is grace day). evaluationEnd = Tuesday (yesterday). Loop fires once → 1 violation. weeklyViolationCount = 1. Threshold = 2 → no recovery yet.

DevOptions: jump to Thursday (skip Wednesday)

Recovery mode active ✓

Note: If you jump multiple days at once (e.g., Monday → Friday in one DevOptions action), all intermediate days are still caught by the catch-up loop. Recovery will still trigger on the same tick as long as 2 missed days are covered in that single jump.

5 Execution Trace — Session Mode (Fresh Install, Tuesday Creation)

Day 0 — Protocol Created (Tuesday 10:00 AM → normalized to Tuesday 00:00)

createdAt = Tuesday 00:00. weekInterval.start = Monday 00:00. Grace condition: Tuesday 00:00 > Monday 00:00 = TRUE → full-week grace granted.

DevOptions: skip to Thursday, Friday, Saturday — no completions

Tick fires each time.

evaluateDailyCompliance — skipped (session protocol)

Returns early. No violations.

evaluateWeekCatchUp — skipped (week not over)

Creation week ends next Monday. All three ticks skip this entirely.

evaluateIntraWeekSessionShortfallIfNeeded — suppressed by grace

Calls evaluateInitialPartialWeeklyGrace → shouldSuppressShortfall = TRUE. Returns early. Zero violations.

Monday of Week 2 — week catch-up processes creation week

Now the creation week is fully ended. evaluateWeekCatchUp evaluates it — but calls the same grace check → still suppressed. Zero violations for the first week.

User skips all of Week 2 — no completions

End of Week 2: evaluateWeekCatchUp evaluates Week 2. No grace for Week 2. Calls countCompletions() — but window boundary may now miscount (Bug #4). If count is 0 and target is 1+, violation written. Count = 1 ≥ threshold 1 → recovery triggered.

Result: 14+ days of complete non-compliance with zero violations until Week 2 ends

Recovery mode requires waiting for two full weeks to pass from a Tuesday creation.

6 Redesign Proposal — Observation-Based Engine

💡 Core shift: Derive violation status directly from completion records at evaluation time. Replace accumulating counters (weeklyViolationCount) with a single idempotent reconciliation pass that checks all past periods on every tick. This eliminates all suppression chains, makes the system predictable, and solves the window-boundary bug structurally.

Proposed Data Model Change

Long-term, remove weeklyViolationCount: Int and weeksEvaluated: Set<WeekID> from Window. Derive all violation counts from violations at read time. In the phased implementation below, remove dependence on weeklyViolationCount first, but keep weeksEvaluated until the incremental weekly evaluator is replaced by the reconciliation engine.

// Window.swift — remove these two fields:
// var weeklyViolationCount: Int      ← DELETE
// var weeksEvaluated: Set<WeekID>   ← DELETE only after Phase 3 removes the incremental evaluator

// NonNegotiable+Computed.swift — add derived properties:
extension NonNegotiable {
    func violationCount(inWindow windowIndex: Int) -> Int {
        violations.filter { $0.windowIndex == windowIndex }.count

    var activeWindowViolationCount: Int {
        guard let wi = windows.indices.last else { return 0 }

Proposed: Single Reconciliation Pass

Replace the three separate evaluation methods with one reconcileViolations that processes all past periods idempotently:

// Replaces evaluateDailyCompliance + evaluateWeekCatchUp + evaluateIntraWeekSessionShortfallIfNeeded
func reconcileViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    guard nn.state == .active || nn.state == .recovery else { return }
    switch nn.definition.mode {
    case .daily:   reconcileDailyViolations(&nn, referenceDate: referenceDate)
    case .session: reconcileSessionViolations(&nn, referenceDate: referenceDate)
    }
    updateRecoveryStateFromViolations(&nn)
}

func reconcileDailyViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    let startOfToday = DateRules.startOfDay(referenceDate, calendar: calendar)
    let lockStart    = DateRules.startOfDay(nn.lock.startDate, calendar: calendar)
    let graceCutoff  = calendar.date(byAdding: .day, value: 1, to: lockStart)!

    var day = lockStart
    while day < startOfToday {
        defer { day = calendar.date(byAdding: .day, value: 1, to: day)! }
        if day < graceCutoff { continue }   // creation day grace
        let dayId = DayIdentifier(day)
        if nn.violations.contains(where: { $0.dayId == dayId }) { continue }  // idempotent
        let hadCompletion = nn.completions.contains {
            $0.kind == .counted && calendar.isDate($0.date, inSameDayAs: day)
        }
        if !hadCompletion {
            let wi = windowIndex(for: nn, date: day) ?? nn.windows.indices.last!
            nn.violations.append(Violation(date: day, kind: .missedDailyCompliance, windowIndex: wi,
                                           weekId: DateRules.weekID(for: day, calendar: calendar)))
        }
    }
}

func reconcileSessionViolations(_ nn: inout NonNegotiable, referenceDate: Date) {
    let startOfToday  = DateRules.startOfDay(referenceDate, calendar: calendar)
    let lockStart      = DateRules.startOfDay(nn.lock.startDate, calendar: calendar)
    let creationWeekId = DateRules.weekID(for: lockStart, calendar: calendar)
    let expected       = expectedCompletionsPerWeek(for: nn.definition)

    var weekStart = DateRules.weekInterval(containing: lockStart, calendar: calendar).start
    while weekStart < startOfToday {
        let weekEnd = calendar.date(byAdding: .day, value: 7, to: weekStart)!
        let weekId  = DateRules.weekID(for: weekStart, calendar: calendar)
        defer { weekStart = weekEnd }

        if nn.violations.contains(where: { $0.weekId == weekId }) { continue }  // idempotent

        // Proportional creation-week grace: only suppress if genuinely couldn't hit target
        if weekId == creationWeekId {
            let feasible = feasibleDaysInWeek(weekStart: weekStart, weekEnd: weekEnd,
                                               createdAt: lockStart, calendar: calendar)
            if feasible < expected { continue }
        }

        // Count completions for the full ISO week — no window boundary filter
        let completions = nn.completions.filter {
            $0.kind == .counted && $0.date >= weekStart && $0.date < weekEnd
        }.count

        let weekDone        = weekEnd <= startOfToday
        let feasibleRemaining = feasibleDaysRemaining(from: startOfToday, weekEnd: weekEnd, lock: nn.lock)
        let impossible      = completions + feasibleRemaining < expected

        if (weekDone || impossible) && completions < expected {
            // Assign violation to the window active at the end of the compliance period
            let assignDate = weekDone ? calendar.date(byAdding: .second, value: -1, to: weekEnd)! : startOfToday
            let wi = windowIndex(for: nn, date: assignDate) ?? nn.windows.indices.last!
            nn.violations.append(Violation(date: assignDate, kind: .missedWeeklyFrequency,
                                           windowIndex: wi, weekId: weekId))
        }
    }
}

private func feasibleDaysInWeek(weekStart: Date, weekEnd: Date,
                                  createdAt: Date, calendar: Calendar) -> Int {
    let effectiveStart = max(weekStart, createdAt)
    var count = 0, d = effectiveStart
    while d < weekEnd {
        count += 1
        d = calendar.date(byAdding: .day, value: 1, to: d)!
    }
    return count
}
✅ Key properties of this design: (1) Idempotent — re-running produces the same violation list. (2) Counts completions per full ISO week without a window boundary filter, fixing Bug #4. (3) Assigns the violation to whatever window was active at the end of the compliance period, not the start. (4) Graceful with DevOptions — skipping N days produces all N violations in one tick.
