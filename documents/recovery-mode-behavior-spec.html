1. Purpose

Recovery Mode is a structured, time-limited protective state that the system enters when a user has accumulated enough missed commitments (violations) on one or more protocols to indicate that their current load is unsustainable. Its goals are:

  • Protect against compounding failure. Without intervention, one struggling protocol generates cascading violations, damages reliability scores, and demoralises the user. Recovery Mode interrupts that spiral.
  • Force an explicit load decision. The user must consciously acknowledge the problem and, where capacity allows, choose one protocol to temporarily suspend. This prevents passive avoidance.
  • Provide a measurable exit condition. Seven consecutive clean days — each with at least one tracked completion (or an explicit exemption) and no new violations — restore the system to normal operation. The exit is earned, not automatic.
  • Keep the plan layer coherent. Future plan allocations for paused protocols are put on hold during recovery and are either restored or marked as terminally skipped on exit, so the plan remains accurate.
  • Enforce policy guardrails during recovery. Creating new protocols and acting on suspended protocols are blocked until recovery resolves.
What Recovery Mode is not: it is not a punishment or a reset. Protocols stay in the system. Completions still count. Reliability scores still update. Recovery is a temporary triage mode that acknowledges reduced capacity, not a wipe of prior progress.

3. Intended Recovery Mode Flow

1

Entering Recovery — Violation Threshold Crossed

During the daily integrity tick (runDailyIntegrityTick), NonNegotiableEngine.evaluateDailyComplianceIfNeeded and evaluateWeekIfNeeded append violations for missed days or missed weekly frequencies. After each append, updateRecoveryState checks the current 14-day window. When a protocol's window violation count reaches or exceeds threshold, its state transitions to .recovery.

2

Entering Recovery — System Transition Detected

CommitmentSystemStore.applySystemUpdate calls handleRecoveryTransition, which detects the first moment a protocol enters .recovery when the system was not already in recovery. It sets:

  • recoveryEntryPendingResolution = true
  • recoveryEntryRequiresPauseSelection = true if there is at least one other .active or .recovery protocol available to pause.
  • recoveryEntryTriggerProtocolId = ID of the triggering protocol.
  • Clears any stale recoveryPausedProtocolId.
  • Resets recoveryCleanDayStreak to 0.
  • Backdates lastRecoveryEvaluationDay to the day before entry.
3

Presenting the Recovery Entry Popup

MainAppView.evaluateRecoveryEntryPresentation observes store state and calls recoveryEntryContext(). If a non-nil context is returned (pending resolution exists), it sets AppRouter.presentRecoveryEntry = true, which renders the RecoveryModePopup as a full-screen modal overlay above the app shell.

4

Selecting or Pausing a Protocol

RecoveryModeViewModel.refresh builds ranked RecoveryProtocolOption items using current-window violations, planned weekly load, and active planned load this week. The popup moves through three steps:

  • Entry step — describes what recovery means and which protocol triggered it.
  • Select Protocol step — shown only if recoveryEntryRequiresPauseSelection is true. The user picks one protocol to suspend.
  • Confirmed step — summary screen before the user dismisses.

If selection is required, confirmPauseSelection calls CommitmentSystemStore.pauseProtocolForRecovery (which sets the protocol state to .suspended and records recoveryPausedProtocolId) and immediately calls PlanStore.pauseAllocations (which sets all future active allocations for that protocol to .paused).

5

Behavior During Recovery Mode

While any protocol is in .recovery state:

  • CommitmentPolicyEngine.canCreate → .deny(.recoveryActive). No new protocols may be created.
  • CommitmentPolicyEngine.canRecordCompletion and canPlaceAllocation → .deny(.protocolSuspended) for the paused protocol.
  • PlanStore.shouldApplyRecoveryStrictness returns true, reducing placement freedom for manual scheduling.
  • PlanRegulatorEngine sorts recovery protocols to the top of the scheduling queue and applies a same-day isolation penalty to avoid co-scheduling recovery and heavy active protocols.
  • The Cockpit shows a recovery progress bar (recoveryProgressText, recoveryProgressFill) and suspended badge on the paused protocol card.
  • Daily Check-In tracks recovery and active protocols but blocks completion recording for suspended ones (systemModeLabel, effectiveWeeklyTarget).
6

Clean-Day Counting

Each day, CommitmentSystemEngine.evaluateRecoveryDay evaluates the most recently closed day. A day qualifies as clean when:

  • No new violation was recorded that day, and
  • Either: at least one counted completion exists across all .active or .recovery protocols, or the day is exempt because no counted completion was required (grace window, weekly-feasibility logic via requiresCountedCompletionForRecovery).

On a clean day, recoveryCleanDayStreak increments. On a violation day, it resets to 0.

7

Exiting Recovery — Normal Path (7 Clean Days)

When recoveryCleanDayStreak reaches 7, evaluateRecoveryDay exits recovery: all .recovery protocols transition to .active, resumeSuspendedIfCapacityAllows restores the paused (.suspended) protocol to .active if the new active count is within capacity, and recoveryCleanDayStreak resets.

8

Exiting Recovery — Normalization Path (Early Exit)

CommitmentSystemEngine.normalizeRecoveryDomain exits recovery early if fewer than 2 recovery-eligible protocols remain — for example, because a protocol was retired during recovery. This path clears stale paused IDs and pending-entry flags as part of the normalization pass (RecoveryNormalizationDecision).

9

Finalizing Allocations on Recovery Exit

When MainAppView detects that the system has exited recovery (no protocol is in .recovery state), it calls PlanStore.finalizeRecoveryAllocationStatuses with the set of protocol IDs that should be restored (activeProtocolIdsForRecoveryRestoration). The logic:

  • Future .paused allocations for restorable protocols → .active
  • Past or non-restorable .paused allocations → .skippedDueToRecovery
10

Cleaning Up Stale State

recomputeRecoveryEntryPendingDecision is called after certain mutations (e.g., protocol retirement) to repair or clear pending-entry state that may have become invalid. It checks whether the trigger protocol still exists, whether candidates remain, and whether the pending flag should be closed automatically.

5. Required Invariants

  • I1
    recoveryPausedProtocolId must never reference a protocol that is retired, deleted, or not present in CommitmentSystem.nonNegotiables. If the referenced protocol disappears, this field must be cleared immediately.
  • I2
    recoveryEntryPendingResolution must be cleared before recovery can exit via any path. A system where recovery has exited but recoveryEntryPendingResolution is still true is corrupt state.
  • I3
    When recovery exits (normal or normalization path), all protocols that were in .recovery state must transition to .active in the same atomic update. No protocol may linger in .recovery after the exit pass.
  • I4
    When recovery exits, PlanStore.finalizeRecoveryAllocationStatuses must be called exactly once. Calling it zero times leaves plan allocations permanently paused; calling it more than once risks flipping already-resolved statuses.
  • I5
    recoveryCleanDayStreak must never exceed 7. Upon reaching 7 it triggers exit; it does not continue counting.
  • I6
    A protocol in .suspended state must always have a corresponding recoveryPausedProtocolId entry. If recoveryPausedProtocolId is nil while any protocol is in .suspended state, the reference is orphaned and must be repaired.
  • I7
    Weekly quota violations must not be counted against a protocol that is currently in .suspended state. Suspended protocols are excluded from active commitment tracking for violation purposes.
  • I8
    CommitmentPolicyEngine.isSystemStable must return false whenever recoveryEntryPendingResolution is true. Unstable system state must block plan creation, not just new protocol creation.
  • I9
    The pause action must be atomic across both stores: CommitmentSystemStore.pauseProtocolForRecovery and PlanStore.pauseAllocations must both succeed or neither must apply. Partial execution (one store updated, the other not) is invalid state.
  • I10
    Only one protocol may be in .suspended state at any time. If normalization or a code path produces multiple suspended protocols, only the one referenced by recoveryPausedProtocolId is valid; the rest must be restored to their prior state.
  • I11
    On app restart, if recoveryEntryPendingResolution is true and the trigger protocol ID references a valid, non-retired protocol, the recovery popup must be re-presented on the next scene activation.
  • I12
    The lastRecoveryEvaluationDay must never advance past today's date (as reported by AppClock). Evaluation of future days is not permitted.

6. Edge Cases

EC-1 — Paused Protocol Retires During Recovery

If the user retires the protocol stored in recoveryPausedProtocolId while the system is in recovery mode:

  • normalizeRecoveryDomain detects that the suspended protocol has become .retired.
  • It must clear recoveryPausedProtocolId and call resumeSuspendedIfCapacityAllows (no-op since the protocol is retired).
  • All plan allocations for that protocol that are currently .paused must be marked .skippedDueToRecovery immediately, not deferred until exit.
  • Recovery itself may continue if the required minimum recovering protocols still remain.
Gap: current code's normalization path clears the ID but the allocation finalization may not fire immediately on retirement — only on full recovery exit. This is Gap G-5.

EC-2 — No Active Protocols Remain After Pause

If, after the user selects a protocol to pause, all remaining non-paused protocols are in .recovery state and none are .active:

  • The system is valid but extremely constrained. Policy blocks creation; the regulator has only recovery protocols to schedule.
  • Clean-day evaluation still works: any counted completion on any .recovery protocol counts.
  • Proposed The Cockpit should render a specific empty-state message indicating no active protocols during recovery.

EC-3 — App Restart During Recovery

All recovery state is persisted (CommitmentSystem is JSON-serialized). On restart:

  • All recovery fields are restored from disk.
  • If recoveryEntryPendingResolution is true, the popup must re-evaluate and re-present on the first scene activation.
  • lastRecoveryEvaluationDay is used to catch up missed days of clean-day evaluation without re-evaluating already-processed days.
  • Invariant I11 must hold.

EC-4 — Trigger Protocol Deleted During Recovery

If recoveryEntryTriggerProtocolId references a protocol that is retired between entry and resolution:

  • recomputeRecoveryEntryPendingDecision must detect the invalid reference.
  • If the system is still in recovery (other protocols in .recovery), it must find a new trigger reference or proceed without one.
  • If no .recovery protocols remain, the pending entry must be auto-closed.

EC-5 — User Exits Recovery Popup With Incomplete Plan

The popup can only be dismissed via the Confirmed step's "Done" button (which calls completeFlow). There is no back-out path that leaves recoveryEntryPendingResolution = true without a pause decision recorded. However:

  • Proposed If the user force-quits the app mid-popup, the persisted state still has recoveryEntryPendingResolution = true but no recoveryPausedProtocolId. On next launch the popup re-presents at the entry step, which is correct behavior.
  • The app must not allow navigation away from the popup before the flow is complete (modal, not sheet-dismissable by swipe).

EC-6 — Stale Recovery State From Previous App Versions

Legacy Codable init(from:) uses decodeIfPresent + fallback for all recovery fields. The fallback values must be safe defaults:

  • recoveryEntryPendingResolution defaults to false — correct.
  • recoveryPausedProtocolId defaults to nil — correct.
  • recoveryCleanDayStreak defaults to 0 — correct.
  • Proposed A post-decode normalization pass should validate that any decoded recoveryPausedProtocolId still references a valid, non-retired protocol; if not, clear it.

EC-7 — Quota-Based Weekly Protocols During Recovery

A protocol with a weekly frequency target (not daily) can still accumulate .missedWeeklyFrequency violations during recovery. If the suspended protocol was weekly-frequency-based:

  • Violations must not be recorded against a .suspended protocol (Invariant I7).
  • effectiveWeeklyTarget in the Daily Check-In ViewModel must exclude suspended protocols from total-quota calculations.
  • When the protocol is restored on recovery exit, the remaining days in the current week may be too few to achieve the weekly target. The grace / feasibility logic in requiresCountedCompletionForRecovery and feasibleCompletionDays must account for this so the first partial week after restoration does not immediately re-trigger recovery.

8. Recommended Implementation Order

Steps are sized for individual Codex / Gemini / Claude tasks. Each step should be a single commit or PR.

Phase A — Correctness Fixes (P0)

  1. A1 — Fix candidate filter in recoveryEntryContext (Gap G-1).
    Change the candidate list to only include protocols in .active state. Update any popup copy that says "active protocol" to match the enforced behavior. Add a simulation assertion that no .recovery protocol appears in the candidate list.
  2. A2 — Introduce a single-call coordinator for the pause action (Gap G-2).
    Move the two-store pause sequence from RecoveryModeViewModel.confirmPauseSelection into a new function on CommitmentSystemStore. The ViewModel calls one function; both stores are updated inside it. Add Invariant I9 guard.
  3. A3 — Add Invariant I10 guard in pauseProtocolForRecovery (Gap G-10).
    One-liner precondition. Can be done alongside A2.

Phase B — Correctness Fixes (P1)

  1. B1 — Resolve or remove unused referenceDate parameters (Gap G-3).
    Decide: remove the parameter and use AppClock.now, or implement it. Whichever is chosen, the API must match what callers can rely on.
  2. B2 — Add post-decode validation for recoveryPausedProtocolId (Gap G-8).
    Inside CommitmentSystem.init(from:), after decoding, check the ID against the decoded protocol list. Clear if not found. Write a migration test using a hand-crafted JSON fixture with a stale ID.
  3. B3 — Finalize allocations immediately on retired-paused-protocol retirement (Gap G-5).
    In the normalization path that clears a retired paused ID, call PlanStore.finalizeRecoveryAllocationStatuses for that specific ID with an empty restorable set (all become .skippedDueToRecovery).
  4. B4 — Fix weekly-frequency post-restoration feasibility (Gap G-9).
    Extend requiresCountedCompletionForRecovery / feasibleCompletionDays to detect a partial week at restoration and pro-rate or exempt accordingly. Write a simulation that restores a weekly protocol on Thursday and confirms no immediate violation at week-end.

Phase C — Coverage & Observability (P1 / P2)

  1. C1 — Add simulation coverage for the popup presentation path (Gap G-4).
    Write assertions in a new runRecoveryPopupPresentationSimulation function: context returns non-nil when pending, returns nil after resolution, re-presents after a simulated restart.
  2. C2 — Add simulation coverage for RecoveryModeViewModel ranking (Gap G-6).
    Write assertions that the highest-violation protocol ranks first, load tiebreakers work, and an empty candidate list is safe.
  3. C3 — Add simulation coverage for recovery placement strictness (Gap G-7).
    Assert shouldApplyRecoveryStrictness returns true when a protocol is in .recovery, and false when all are active.
