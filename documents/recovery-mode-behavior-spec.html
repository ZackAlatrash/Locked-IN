1. Purpose

Recovery Mode is a structured, time-limited protective state that the system enters when a user has accumulated enough missed commitments (violations) on one or more protocols to indicate that their current load is unsustainable. Its goals are:

  • Protect against compounding failure. Without intervention, one struggling protocol generates cascading violations, damages reliability scores, and demoralises the user. Recovery Mode interrupts that spiral.
  • Force an explicit load decision. The user must consciously acknowledge the problem and, where capacity allows, choose one protocol to temporarily suspend. This prevents passive avoidance.
  • Provide a measurable exit condition. Seven consecutive clean days — each with at least one tracked completion (or an explicit exemption) and no new violations — restore the system to normal operation. The exit is earned, not automatic.
  • Keep the plan layer coherent. Future plan allocations for paused protocols are put on hold during recovery and are either restored or marked as terminally skipped on exit, so the plan remains accurate.
  • Enforce policy guardrails during recovery. Creating new protocols and acting on suspended protocols are blocked until recovery resolves.
What Recovery Mode is not: it is not a punishment or a reset. Protocols stay in the system. Completions still count. Reliability scores still update. Recovery is a temporary triage mode that acknowledges reduced capacity, not a wipe of prior progress.

2. Current Implementation Summary

  • NonNegotiableState.swift — enum with .recovery, .suspended, .retired, .active.

3. Intended Recovery Mode Flow

1

Entering Recovery — Violation Threshold Crossed

During the daily integrity tick (runDailyIntegrityTick), NonNegotiableEngine.evaluateDailyComplianceIfNeeded and evaluateWeekIfNeeded append violations for missed days or missed weekly frequencies. After each append, updateRecoveryState checks the current 14-day window. When a protocol's window violation count reaches or exceeds threshold, its state transitions to .recovery.

2

Entering Recovery — System Transition Detected

CommitmentSystemStore.applySystemUpdate calls handleRecoveryTransition, which detects the first moment a protocol enters .recovery when the system was not already in recovery. It sets:

  • recoveryEntryPendingResolution = true
  • recoveryEntryRequiresPauseSelection = true if there is at least one other .active or .recovery protocol available to pause.
  • recoveryEntryTriggerProtocolId = ID of the triggering protocol.
  • Clears any stale recoveryPausedProtocolId.
  • Resets recoveryCleanDayStreak to 0.
  • Backdates lastRecoveryEvaluationDay to the day before entry.
3

Presenting the Recovery Entry Popup

MainAppView.evaluateRecoveryEntryPresentation observes store state and calls recoveryEntryContext(). If a non-nil context is returned (pending resolution exists), it sets AppRouter.presentRecoveryEntry = true, which renders the RecoveryModePopup as a full-screen modal overlay above the app shell.

4

Selecting or Pausing a Protocol

RecoveryModeViewModel.refresh builds ranked RecoveryProtocolOption items using current-window violations, planned weekly load, and active planned load this week. The popup moves through three steps:

  • Entry step — describes what recovery means and which protocol triggered it.
  • Select Protocol step — shown only if recoveryEntryRequiresPauseSelection is true. The user picks one protocol to suspend.
  • Confirmed step — summary screen before the user dismisses.

If selection is required, confirmPauseSelection calls CommitmentSystemStore.pauseProtocolForRecovery (which sets the protocol state to .suspended and records recoveryPausedProtocolId) and immediately calls PlanStore.pauseAllocations (which sets all future active allocations for that protocol to .paused).

5

Behavior During Recovery Mode

While any protocol is in .recovery state:

  • CommitmentPolicyEngine.canCreate → .deny(.recoveryActive). No new protocols may be created.
  • CommitmentPolicyEngine.canRecordCompletion and canPlaceAllocation → .deny(.protocolSuspended) for the paused protocol.
  • PlanStore.shouldApplyRecoveryStrictness returns true, reducing placement freedom for manual scheduling.
  • PlanRegulatorEngine sorts recovery protocols to the top of the scheduling queue and applies a same-day isolation penalty to avoid co-scheduling recovery and heavy active protocols.
  • The Cockpit shows a recovery progress bar (recoveryProgressText, recoveryProgressFill) and suspended badge on the paused protocol card.
  • Daily Check-In tracks recovery and active protocols but blocks completion recording for suspended ones (systemModeLabel, effectiveWeeklyTarget).
6

Clean-Day Counting

Each day, CommitmentSystemEngine.evaluateRecoveryDay evaluates the most recently closed day. A day qualifies as clean when:

  • No new violation was recorded that day, and
  • Either: at least one counted completion exists across all .active or .recovery protocols, or the day is exempt because no counted completion was required (grace window, weekly-feasibility logic via requiresCountedCompletionForRecovery).

On a clean day, recoveryCleanDayStreak increments. On a violation day, it resets to 0.

7

Exiting Recovery — Normal Path (7 Clean Days)

When recoveryCleanDayStreak reaches 7, evaluateRecoveryDay exits recovery: all .recovery protocols transition to .active, resumeSuspendedIfCapacityAllows restores the paused (.suspended) protocol to .active if the new active count is within capacity, and recoveryCleanDayStreak resets.

8

Exiting Recovery — Normalization Path (Early Exit)

CommitmentSystemEngine.normalizeRecoveryDomain exits recovery early if fewer than 2 recovery-eligible protocols remain — for example, because a protocol was retired during recovery. This path clears stale paused IDs and pending-entry flags as part of the normalization pass (RecoveryNormalizationDecision).

9

Finalizing Allocations on Recovery Exit

When MainAppView detects that the system has exited recovery (no protocol is in .recovery state), it calls PlanStore.finalizeRecoveryAllocationStatuses with the set of protocol IDs that should be restored (activeProtocolIdsForRecoveryRestoration). The logic:

  • Future .paused allocations for restorable protocols → .active
  • Past or non-restorable .paused allocations → .skippedDueToRecovery
10

Cleaning Up Stale State

recomputeRecoveryEntryPendingDecision is called after certain mutations (e.g., protocol retirement) to repair or clear pending-entry state that may have become invalid. It checks whether the trigger protocol still exists, whether candidates remain, and whether the pending flag should be closed automatically.

5. Required Invariants

  • I1
    recoveryPausedProtocolId must never reference a protocol that is retired, deleted, or not present in CommitmentSystem.nonNegotiables. If the referenced protocol disappears, this field must be cleared immediately.
  • I2
    recoveryEntryPendingResolution must be cleared before recovery can exit via any path. A system where recovery has exited but recoveryEntryPendingResolution is still true is corrupt state.
  • I3
    When recovery exits (normal or normalization path), all protocols that were in .recovery state must transition to .active in the same atomic update. No protocol may linger in .recovery after the exit pass.
  • I4
    When recovery exits, PlanStore.finalizeRecoveryAllocationStatuses must be called exactly once. Calling it zero times leaves plan allocations permanently paused; calling it more than once risks flipping already-resolved statuses.
  • I5
    recoveryCleanDayStreak must never exceed 7. Upon reaching 7 it triggers exit; it does not continue counting.
  • I6
    A protocol in .suspended state must always have a corresponding recoveryPausedProtocolId entry. If recoveryPausedProtocolId is nil while any protocol is in .suspended state, the reference is orphaned and must be repaired.
  • I7
    Weekly quota violations must not be counted against a protocol that is currently in .suspended state. Suspended protocols are excluded from active commitment tracking for violation purposes.
  • I8
    CommitmentPolicyEngine.isSystemStable must return false whenever recoveryEntryPendingResolution is true. Unstable system state must block plan creation, not just new protocol creation.
  • I9
    The pause action must be atomic across both stores: CommitmentSystemStore.pauseProtocolForRecovery and PlanStore.pauseAllocations must both succeed or neither must apply. Partial execution (one store updated, the other not) is invalid state.
  • I10
    Only one protocol may be in .suspended state at any time. If normalization or a code path produces multiple suspended protocols, only the one referenced by recoveryPausedProtocolId is valid; the rest must be restored to their prior state.
  • I11
    On app restart, if recoveryEntryPendingResolution is true and the trigger protocol ID references a valid, non-retired protocol, the recovery popup must be re-presented on the next scene activation.
  • I12
    The lastRecoveryEvaluationDay must never advance past today's date (as reported by AppClock). Evaluation of future days is not permitted.

6. Edge Cases

EC-1 — Paused Protocol Retires During Recovery

If the user retires the protocol stored in recoveryPausedProtocolId while the system is in recovery mode:

  • normalizeRecoveryDomain detects that the suspended protocol has become .retired.
  • It must clear recoveryPausedProtocolId and call resumeSuspendedIfCapacityAllows (no-op since the protocol is retired).
  • All plan allocations for that protocol that are currently .paused must be marked .skippedDueToRecovery immediately, not deferred until exit.
  • Recovery itself may continue if the required minimum recovering protocols still remain.
Gap: current code's normalization path clears the ID but the allocation finalization may not fire immediately on retirement — only on full recovery exit. This is Gap G-5.

EC-2 — No Active Protocols Remain After Pause

If, after the user selects a protocol to pause, all remaining non-paused protocols are in .recovery state and none are .active:

  • The system is valid but extremely constrained. Policy blocks creation; the regulator has only recovery protocols to schedule.
  • Clean-day evaluation still works: any counted completion on any .recovery protocol counts.
  • Proposed The Cockpit should render a specific empty-state message indicating no active protocols during recovery.

EC-3 — App Restart During Recovery

All recovery state is persisted (CommitmentSystem is JSON-serialized). On restart:

  • All recovery fields are restored from disk.
  • If recoveryEntryPendingResolution is true, the popup must re-evaluate and re-present on the first scene activation.
  • lastRecoveryEvaluationDay is used to catch up missed days of clean-day evaluation without re-evaluating already-processed days.
  • Invariant I11 must hold.

EC-4 — Trigger Protocol Deleted During Recovery

If recoveryEntryTriggerProtocolId references a protocol that is retired between entry and resolution:

  • recomputeRecoveryEntryPendingDecision must detect the invalid reference.
  • If the system is still in recovery (other protocols in .recovery), it must find a new trigger reference or proceed without one.
  • If no .recovery protocols remain, the pending entry must be auto-closed.

EC-5 — User Exits Recovery Popup With Incomplete Plan

The popup can only be dismissed via the Confirmed step's "Done" button (which calls completeFlow). There is no back-out path that leaves recoveryEntryPendingResolution = true without a pause decision recorded. However:

  • Proposed If the user force-quits the app mid-popup, the persisted state still has recoveryEntryPendingResolution = true but no recoveryPausedProtocolId. On next launch the popup re-presents at the entry step, which is correct behavior.
  • The app must not allow navigation away from the popup before the flow is complete (modal, not sheet-dismissable by swipe).

EC-6 — Stale Recovery State From Previous App Versions

Legacy Codable init(from:) uses decodeIfPresent + fallback for all recovery fields. The fallback values must be safe defaults:

  • recoveryEntryPendingResolution defaults to false — correct.
  • recoveryPausedProtocolId defaults to nil — correct.
  • recoveryCleanDayStreak defaults to 0 — correct.
  • Proposed A post-decode normalization pass should validate that any decoded recoveryPausedProtocolId still references a valid, non-retired protocol; if not, clear it.

EC-7 — Quota-Based Weekly Protocols During Recovery

A protocol with a weekly frequency target (not daily) can still accumulate .missedWeeklyFrequency violations during recovery. If the suspended protocol was weekly-frequency-based:

  • Violations must not be recorded against a .suspended protocol (Invariant I7).
  • effectiveWeeklyTarget in the Daily Check-In ViewModel must exclude suspended protocols from total-quota calculations.
  • When the protocol is restored on recovery exit, the remaining days in the current week may be too few to achieve the weekly target. The grace / feasibility logic in requiresCountedCompletionForRecovery and feasibleCompletionDays must account for this so the first partial week after restoration does not immediately re-trigger recovery.

New Capabilities Not in Original Spec

  • Recovery Scoring Algorithm: RecoveryModeViewModel ranks pause candidates using a weighted formula — violations in current window × 100, weekly load × 10, planned load this week × 1. First option is automatically flagged as recommended.
  • Post-Restoration Week Grace: isPostRestorationPartialWeek() in CommitmentSystemEngine proportionally reduces the effective weekly target for the restoration week so the first partial week does not immediately re-trigger recovery.
  • Capacity Enforcement Bypass: enforceCapacityIfNeeded() short-circuits when recovery entry is pending, preventing capacity pressure from suspending additional protocols while the user has not yet resolved the popup.
  • Two-Overload Plan Finalization: PlanStore.finalizeRecoveryAllocationStatuses has two overloads — one for full exit (takes a set of restorable IDs) and one for single-protocol retirement during recovery.
  • Violation-Based Candidate Ranking: Candidates in the pause selection step show violation counts and planned weekly load, giving the user meaningful data to guide their decision.

EC-8 — Two Protocols Cross Recovery Threshold in the Same Tick

If two protocols both accumulate enough violations to cross the recovery threshold within a single tick (e.g., the app is opened after several missed days and the catch-up loop processes multiple days at once):

  • Both protocols transition to .recovery in the same applySystemUpdate call.
  • handleRecoveryTransition detects the first time the system enters recovery. The trigger protocol ID is set to the first protocol found in the recovery state — determined by array order, not by which protocol hit the threshold first.
  • If the user views the recovery popup, the trigger protocol shown may not be the one with the most violations (it depends on array ordering, not ranking). The candidate list will, however, include all .active protocols for selection.
Recommended fix: In handleRecoveryTransition, when identifying the trigger protocol, prefer the protocol with the highest violation count in its current window rather than using array order. This ensures the most egregious failure is highlighted in the popup entry step.

EC-9 — Recovery Entry While Previous recoveryEntryPendingResolution is Still True

If the system is already in recovery with recoveryEntryPendingResolution = true (user has not dismissed the popup) and an additional protocol crosses the recovery threshold:

  • handleRecoveryTransition checks whether the system was already in recovery before this transition. Since it was, it does not overwrite the trigger protocol ID or reset the pending resolution flag — the existing popup flow continues unchanged.
  • The newly-recovered protocol does not appear as a trigger in the popup, and the user may not realize a second protocol has also entered recovery.
  • The confirmed step will show the full list of protocols in recovery, so this is not entirely invisible — but the entry copy will only reference the original trigger protocol.
Recommended fix: When an additional protocol enters recovery while pending resolution is already active, update the popup's entry copy to reflect that multiple protocols are now in recovery, not just the original trigger. Consider surfacing a count: "2 protocols have entered recovery mode."

EC-10 — Lock Period Expires During Recovery

A protocol's 28/60/90-day lock period ends while the system is in recovery mode. The lock expiry triggers the protocol to complete (.completed or .retired).

  • If the completed protocol was the trigger protocol, recoveryEntryTriggerProtocolId now references a completed protocol. The post-decode validation clears stale recoveryPausedProtocolId references but may not catch a completed (as opposed to retired) trigger protocol.
  • If the completed protocol was the only one in .recovery, normalizeRecoveryDomain should detect fewer than 2 eligible protocols and exit recovery early.
  • If the completed protocol was the paused (.suspended) protocol, recoveryPausedProtocolId would reference a protocol that is now .completed. The guard currently checks for .retired state — it is unclear whether .completed is also treated as invalid.
Recommended fix: Expand the paused-protocol validity check to treat .completed the same as .retired for the purpose of clearing stale references. Verify that normalizeRecoveryDomain and the post-decode validator both handle the .completed case explicitly, not just .retired.

EC-11 — Recovery Exits Mid-Day (Streak Reaches 7 Before Violation Evaluation)

The daily tick runs violation evaluation first, then clean-day evaluation. The order is: evaluateDailyCompliance → evaluateWeekCatchUp → evaluateIntraWeekSession → advanceWindows → evaluateRecoveryDay. This ordering means violations are always recorded before the clean-day streak is checked — correct. However consider the reverse scenario:

  • User is on clean-day streak of 6. App opens on day 7 (the exit day). No violations today. Streak reaches 7 → recovery exits, all protocols transition to .active.
  • Because violation evaluation runs first, any missed days from the previous window are caught. If a violation is found and recorded during the same tick, the streak resets to 0 before the exit check — correct. Recovery does not exit.
  • The ordering is sound. However: after recovery exits in this tick, the violation evaluation stage has already run and found no violations. Later in the same day, if the user opens the app again (scene activation fires another tick), the violation evaluation runs again. For daily mode, lastDailyComplianceCheckedDay prevents double evaluation of the same day — safe. For session mode, the intra-week check is idempotent — safe.
No immediate fix required. The tick ordering is correct and idempotent guards prevent double-evaluation. Document this ordering as an invariant: violation evaluation must always precede clean-day evaluation in the pipeline.

EC-12 — All Protocols Retire During Recovery (Empty System)

The user retires every protocol while in recovery mode — for example, they decide to start fresh and delete everything:

  • As each protocol is retired, applyPostMutationRecoveryNormalization runs normalizeRecoveryDomain. When recovery-eligible count drops below 2, recovery exits.
  • However, if all protocols are retired including the paused one, resumeSuspendedIfCapacityAllows has nothing to restore.
  • PlanStore.finalizeRecoveryAllocationStatuses is called with an empty restorable set — all remaining paused allocations become .skippedDueToRecovery. This is correct.
  • The recovery popup, if still showing, now has no valid trigger protocol. recoveryEntryContext() should return nil because no protocols are in .recovery state, causing the popup to auto-dismiss. Verify this path is covered.
Recommended fix: Add an explicit guard in recoveryEntryContext() that returns nil if nonNegotiables is empty, even if recoveryEntryPendingResolution is still true. Also call clearRecoveryEntryState() in the normalization path when the system becomes fully empty.

EC-13 — Plan Allocation Accumulation During Long Recovery

If recovery lasts many weeks, pauseAllocations() marks all future allocations as .paused at pause-time. However, the plan regulator continues to generate new allocations in subsequent weeks. Those new allocations are created with .active status, not .paused.

  • Newly generated allocations for the suspended protocol after the initial pause are NOT automatically paused — they receive the default .active status.
  • This means the suspended protocol's future plan slots will be partially paused (those that existed at pause time) and partially active (those generated by the regulator afterward), creating an inconsistent plan view.
  • The current implementation of pauseAllocations() only operates at the moment of pause — it is not a live subscription that catches future additions.
Recommended fix: In PlanRegulatorEngine (or when allocations are placed), check if the target protocol is .suspended and if so, create new allocations with .paused status instead of .active. Alternatively, filter suspended protocols out of the regulator's scheduling candidates entirely so no new allocations are generated while suspended.

EC-14 — Threshold Change Retroactively Affects Existing Violation Counts

The violation thresholds were recently lowered (daily: 3→2, session: 2→1). An existing user who had accumulated 2 daily violations in their current window — previously safe under the old threshold of 3 — is now over the new threshold of 2.

  • Recovery entry is triggered only when updateRecoveryState is called, which happens only when a new violation is appended. Old violations that were already recorded do not re-trigger the check.
  • This means the threshold change is NOT retroactive — a user with 2 existing violations in their window stays in .active state until a third violation is recorded (even though the new threshold is 2).
  • This is safe behavior (no surprise recovery entry on app update) but creates a temporary window of inconsistency where a user "should" be in recovery but isn't.
Recommended fix: After any threshold change, run a post-migration integrity pass that calls updateRecoveryState for every active protocol whose current window violation count meets or exceeds the new threshold. Guard this with a UserDefaults migration sentinel (e.g., didRunThresholdMigration20260506) so it only fires once.

EC-15 — Recovery Shortfall Suppression Leaks Into Next Week

When a suspended protocol is restored at recovery exit, recoveryRestoredAt is set on the protocol. isPostRestorationPartialWeek() uses this date to grant a reduced target for the restoration week only — it checks if the recovery week ID matches the current week ID. This is correct.

  • However, if recovery exits late on a Sunday (the last day of the week), the restoration week is week N. One tick later — after midnight, on Monday of week N+1 — the week ID changes. isPostRestorationPartialWeek now compares week N+1 to week N and returns false. Grace expires correctly.
  • The issue is when the tick fires multiple times on Sunday (app backgrounded and foregrounded). Each time, the same partial-week grace applies. The result is the same because the method is idempotent — but if the week boundary is crossed between two ticks on the same session, the grace computation changes mid-session.
  • More critically: the feasibility check in isPostRestorationPartialWeek computes available days from recoveryRestoredAt. If the restored protocol has its weekly target reduced to 0 (restored with zero feasible days), the clean-day evaluation uses effectiveWeeklyTarget = 0, meaning no completion is required that day. But the violation engine also uses this function — if the week then ends with 0 completions and a target of 0, is a violation suppressed? It should be, but verify that both callers handle the zero-target case identically.
Recommended fix: Add an explicit guard in evaluateWeekIfNeeded and reconcileSessionViolations: if effectiveWeeklyTarget == 0, skip violation recording entirely for that week. This makes the zero-target case an unambiguous no-op in both the violation engine and the clean-day engine.

EC-16 — Single-Protocol System: No Candidate to Pause, but Popup Flow Is Not Skipped Completely

When only one protocol exists and it enters recovery, recoveryEntryRequiresPauseSelection = false (there are no other protocols to pause). The popup shows only the entry step and the confirmed step, skipping the select-protocol step. This is correct.

  • However, the clean-day evaluation still requires the single recovering protocol to have at least one completion per day (for daily mode) or hit its weekly target (for session mode) to count a day as clean. With only one protocol in recovery and nothing else active, every missed day resets the streak — potentially trapping the user in permanent recovery if they have a single daily protocol they cannot consistently complete.
  • There is no UX mechanism to communicate that the user's only way out is 7 consecutive days of completing the one protocol — no guidance is shown after the popup closes.
  • There is also no early-exit mechanism for this scenario. The only exits are 7 clean days or normalization (fewer than 2 eligible protocols). A single protocol in recovery is exactly 1 eligible protocol — which is below the <2 threshold, so normalization should exit recovery immediately.
Recommended fix: Audit normalizeRecoveryDomain: if recoveryEligibleCount < 2 and only one protocol has ever been in the system (not retired), verify whether the normalization exit fires correctly. If a single-protocol system can be permanently stuck in recovery without normalization exiting, add a rule: when the system has exactly 1 non-retired protocol and it is in recovery with no suspended protocol, exit recovery automatically (the concept of capacity-load management does not apply to a single protocol).

New — Open During a long recovery, the regulator continues scheduling future weeks. New allocations for the suspended protocol are inserted with .active status (the default). NonNegotiableEngine.updateRecoveryState(), CommitmentSystemStore (app launch path) Add a one-time migration pass (guarded by a UserDefaults sentinel) that calls updateRecoveryState for every active protocol whose current window violation count already meets or exceeds the current threshold. 8. Recommended Implementation Order

Steps are sized for individual Codex / Claude tasks. Each step should be a single commit or PR. Gaps marked Resolved still need simulation coverage added when a test target exists.

Phase A — Correctness Fixes (P0)

  1. A1 — Fix candidate filter to exclude .recovery protocols (Gap G-1).
    In recoveryPauseCandidateIds(), return only .active protocols. If the result is empty, set recoveryEntryRequiresPauseSelection = false rather than falling back to .recovery protocols. Update popup entry copy to explicitly say "active protocol." Add assertion: no .recovery protocol in candidate list.
  2. A2 — Add rollback to the two-store pause coordinator (Gap G-2).
    pauseProtocolForRecoveryWithPlanSync() already exists as a coordinator. Add a rollback block: if PlanStore.pauseAllocations() throws or fails, reverse the commitment store change before re-throwing. The ViewModel already catches errors and shows a warning — ensure the rollback leaves both stores in their original state.
  3. A3 — Exclude suspended protocols from the plan regulator's candidate set (Gap G-11).
    In PlanRegulatorEngine, filter out any protocol in .suspended state when building the scheduling candidate list. This prevents new allocations from being generated for suspended protocols, eliminating the .active / .paused inconsistency. Alternative: creation path checks suspension status and forces .paused status on new allocations for suspended protocols.

Phase B — Correctness Fixes (P1)

  1. B1 — Extend post-decode validation for stale recovery IDs (Gap G-8).
    In CommitmentSystem.init(from:), also clear recoveryPausedProtocolId if the referenced protocol is .completed (not just .retired). Add assertion that paused ID ≠ trigger ID. Write a test using a hand-crafted JSON fixture with each stale scenario.
  2. B2 — Finalize allocations atomically on retirement (Gap G-5).
    In the retirement code path (before returning from the retirement action, not in post-mutation normalization), call PlanStore.finalizeRecoveryAllocationStatuses(protocolId:) with an empty restorable set for the retired protocol. Verify no paused allocation remains for a retired protocol after this call.
  3. B3 — Guard zero-target week against false violations (Gap G-9 + EC-15).
    In evaluateWeekIfNeeded and the proposed reconcileSessionViolations, add: guard effectiveWeeklyTarget > 0 else { continue }. Mirror the same guard in the clean-day evaluation (evaluateRecoveryDay). Write a simulation that restores a session protocol with zero feasible days remaining and confirms no violation fires at week-end.
  4. B4 — One-time migration pass for threshold changes (Gap G-12).
    Add a UserDefaults sentinel (e.g., didRunThresholdMigration20260506). On first launch after update, iterate all active protocols and call updateRecoveryState for any whose current window violation count meets or exceeds the current threshold. This brings existing users into recovery mode correctly after a threshold tightening.
  5. B5 — Resolve unused referenceDate parameters (Gap G-3).
    Audit all public recovery methods on CommitmentSystemStore. Either remove the parameter and use AppClock.now internally, or fully implement date injection so simulation tests can rely on it. No mixed API surface permitted.

Phase C — Edge Case Hardening

  1. C1 — Prefer highest-violation protocol as trigger when multiple hit threshold simultaneously (EC-8).
    In handleRecoveryTransition, when setting recoveryEntryTriggerProtocolId, choose the protocol with the highest current-window violation count from among the newly-recovered protocols rather than relying on array order.
  2. C2 — Update popup copy when additional protocols enter recovery mid-flow (EC-9).
    If recoveryEntryPendingResolution is already true and another protocol enters recovery, update the entry step copy to reflect a count of how many protocols are in recovery ("2 protocols have entered recovery mode"). The trigger protocol ID remains unchanged.
  3. C3 — Expand .completed handling in normalization paths (EC-10).
    In normalizeRecoveryDomain and the post-decode validator, treat .completed the same as .retired when checking whether a paused or trigger protocol reference is still valid. Audit all state == .retired checks and replace with a helper isTerminal predicate that covers both states.
  4. C4 — Verify single-protocol normalization exit (EC-16).
    Audit normalizeRecoveryDomain: confirm that a system with exactly 1 non-retired protocol in .recovery (and no suspended protocol) correctly exits recovery via the recoveryEligibleCount < 2 path. Add a simulation case for this scenario.
  5. C5 — Empty-system guard in recoveryEntryContext() (EC-12).
    Add a guard that returns nil when system.nonNegotiables is empty, regardless of the value of recoveryEntryPendingResolution. Call clearRecoveryEntryState() in normalization when the system becomes fully empty.

Phase D — Coverage & Observability (P1 / P2)

  1. D1 — Add simulation coverage for popup presentation path (Gap G-4).
    Write assertions: context returns non-nil when pending, nil after resolution, re-presents after simulated restart. Convert to XCTest when test target exists.
  2. D2 — Add simulation coverage for RecoveryModeViewModel ranking (Gap G-6).
    Assert highest-violation protocol ranks first, tiebreakers work, empty candidate list is safe.
  3. D3 — Add simulation coverage for recovery placement strictness (Gap G-7).
    Assert shouldApplyRecoveryStrictness returns true in recovery and false when all protocols are active.
