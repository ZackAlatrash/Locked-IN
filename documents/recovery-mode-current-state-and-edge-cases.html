Recovery Mode Current State and Edge Cases

This document compares the current recovery-mode implementation against the intended enhanced behavior described in documents/recovery-mode-behavior-spec.html. It uses the current Swift code as source of truth and lists the edge cases still worth adding, with a recommended fix for each.

Project: Locked IN Date: 2026-05-06 Source: code review + existing recovery docs Status: current-state audit
Bottom line: recovery mode is much stronger than the old behavior spec suggests. Several old P0/P1 gaps are already fixed. The remaining risk is less about the main happy path and more about catch-up evaluation, stale persisted state, atomicity between stores, and duplicated eligibility logic drifting across engine, store, plan, and check-in surfaces.

1. Current State Snapshot

Good

Entry: protocol-level recovery entry now derives the violation count from violations, not legacy Window.weeklyViolationCount. Current thresholds are daily = 2, session = 1.

Refs: NonNegotiableEngine.updateRecoveryState, recoveryViolationThreshold.

Good

Entry flow: the store detects first entry, sets pending resolution, trigger ID, pause requirement, clears stale paused ID, resets clean-day streak, and backdates evaluation day.

Refs: CommitmentSystemStore.handleRecoveryTransition.

Mixed

Pause flow: there is now a single view-model call into pauseProtocolForRecoveryWithPlanSync, and double-pause is guarded. It is still not a true transaction because domain state is persisted before plan allocation persistence is guaranteed.

Mixed

Exit: seven clean days exits recovery and restores suspended protocols if capacity allows. Plan allocation cleanup is still coordinated by MainAppView, not by the store/domain exit operation itself.

2. Current vs Intended Behavior

Area Intended Enhanced Behavior Current Implementation Assessment Recommended Next Step
Recovery entry threshold Enter when current-window violations cross mode-specific threshold. Implemented using derived violationCount(inWindow:). Current values are daily = 2, session = 1, while the old spec still says daily = 3, session = 2. Implemented Spec stale Update the old spec or add a migration note explaining the new stricter thresholds.
Popup re-presentation Pending recovery resolution must survive restart and re-present. Persisted fields exist; simulations cover context after restart. Presentation depends on MainAppView.evaluateRecoveryEntryPresentation. Mostly implemented Add one UI-level/manual checklist around actual sheet presentation after force quit.
Pause candidates Old plan says candidates should be active only. Current engine allows active candidates first, then multiple recovery protocols as fallback. recoveryPauseCandidateIds returns active protocols when available; if none exist and more than one recovery protocol exists, it returns recovery protocols. Policy mismatch Decide the product rule. Either make active-only strict, or keep recovery fallback and update popup copy/spec to explicitly allow it.
Pause atomicity Domain suspend and plan allocation pause must succeed or fail together. The view model calls one coordinator, but pauseProtocolForRecovery persists before PlanStore.pauseAllocations. pauseAllocations swallows save failures. Partial Make plan pause/finalize operations throwing and move persistence so either both stores save or the domain mutation rolls back.
Clean-day evaluation On restart or skipped days, recovery should catch up all closed days since lastRecoveryEvaluationDay. evaluateRecoveryDay evaluates only yesterday once per call, even if several days were missed. Gap Loop from lastRecoveryEvaluationDay + 1 day through yesterday, applying clean/dirty logic per day.
Recovery exit cleanup When recovery exits, domain state and plan allocation cleanup should be consistent. Domain exit happens in CommitmentSystemEngine; plan cleanup happens later in MainAppView when it notices there are no recovery protocols. Fragile Move recovery-exit finalization into a store-level coordinator that receives PlanStore, or emit an explicit exit event consumed exactly once.
Retired paused protocol Retiring the paused protocol should immediately terminalize its paused allocations. Implemented when retireNonNegotiable(..., planStore:) is called with a plan store. If callers omit planStore, finalization cannot happen. Mostly implemented Require the plan-sync retirement path from UI, and add a defensive repair pass for retired paused allocations on plan refresh.
Post-decode stale IDs Invalid recovery IDs should be repaired on load. Decode clears IDs that point to retired/missing protocols, but it does not require recoveryPausedProtocolId to point to a .suspended protocol, nor does it repair orphaned suspended protocols. Partial Add full recovery-state normalization after decode or immediately after repository load.
Post-restoration weekly feasibility A protocol restored mid-week should not immediately re-enter recovery if the weekly target was impossible. Implemented in domain engine for suspended protocols restored via recoveryRestoredAt. Store/logs/daily-check-in/plan still contain older duplicated initial-grace logic and do not fully mirror post-restoration logic. Domain good, surfaces drift Move weekly target/grace/feasibility calculations to one shared policy/helper and replace duplicate store/view-model implementations.

3. Edge Cases To Add

These are the edge cases I would add before calling recovery mode production-hard. Each case includes the current risk and a recommended fix.
ID Edge Case Current Risk Recommended Fix Priority
REC-EC-01 App is closed for several days while already in recovery. Only yesterday is evaluated, so a 5-day absence counts as one recovery day. Earlier dirty days can be skipped for streak purposes. Change evaluateRecoveryDay into a catch-up loop over each closed day since lastRecoveryEvaluationDay. Add simulation: last evaluated Monday, reopen Friday, mixed Tue/Wed/Thu outcomes produce the correct final streak. P0
REC-EC-02 Recovery catch-up runs after daily/weekly violation catch-up added old violations. If new violations are dated before yesterday, the single-day recovery evaluator may not reset streak for those dirty days. Same catch-up loop as REC-EC-01, using the violation log for each evaluated day. Add test where reopening after 4 days writes violations on day 2 and day 3. P0
REC-EC-03 Domain pause succeeds but plan allocation pause fails to persist. The protocol is .suspended, but future allocations may remain .active, making the plan contradict policy. Make PlanStore.pauseAllocations throw on save failure. Have pauseProtocolForRecoveryWithPlanSync prepare both mutations and persist both, or roll back the domain mutation if plan persistence fails. P0
REC-EC-04 Recovery exits outside MainAppView presentation lifecycle. Paused allocations may remain .paused until a UI pass notices no protocol is in recovery. Move exit cleanup to a store coordinator such as runDailyIntegrityTick(referenceDate:planStore:), or persist a one-time recoveryExitPendingPlanFinalization event. P1
REC-EC-05 Decoded recoveryPausedProtocolId points to an active protocol. Current decode keeps any non-retired ID, so a stale paused ID can survive even when the referenced protocol is not suspended. On decode/load, clear recoveryPausedProtocolId unless the referenced protocol exists and is .suspended. Add fixture coverage. P1
REC-EC-06 A protocol is .suspended but recoveryPausedProtocolId is nil. The invariant is orphaned. Policy blocks the protocol, but recovery exit may not know whether to restore it intentionally. Add a load-time repair rule: if exactly one suspended protocol exists during recovery and paused ID is nil, set it; otherwise mark as corrupt and normalize by restoring or requiring explicit user resolution. P1
REC-EC-07 Decoded lastRecoveryEvaluationDay is in the future because of clock changes or old test data. Recovery clean-day evaluation stalls until real time catches up. Normalize lastRecoveryEvaluationDay to at most yesterday during store load, using AppClock.now or the current reference date. P1
REC-EC-08 Decoded recoveryCleanDayStreak is greater than 7. The value is logically invalid. It may display incorrect progress until the next evaluation mutates it. Clamp to 0...6 while still in recovery, or immediately run exit normalization if the value is >= 7. P1
REC-EC-09 Pause candidate fallback selects a .recovery protocol when no active protocols exist. This can be valid, but it contradicts popup copy that says "active protocol" and old implementation-plan guidance. Choose one rule. If fallback remains, update copy to "choose one protocol" and immediately run normalization after pausing a recovery protocol. If active-only is desired, remove the recovery fallback. P1
REC-EC-10 Pausing a recovery protocol leaves fewer than two recovery-eligible protocols. Early-exit normalization uses recoveryEligibleCount < 2, but it is not called after pause. Recovery can continue in a state that normalization would exit. After any pause, call normalizeRecoveryDomain. If it exits recovery, clear the pending flow and finalize allocations consistently. P1
REC-EC-11 Recovery entry popup remains unresolved for days while DevOptions/background ticks continue. Recovery can progress or exit before the user completes the entry decision. The modal UI normally prevents this, but test/runtime paths can bypass it. While recoveryEntryPendingResolution is true, either pause recovery clean-day progression or require popup resolution before evaluating exit. P1
REC-EC-12 runSystemIntegrityCheck is called instead of runDailyIntegrityTick. It evaluates violations and windows but does not call evaluateRecoveryDay, so recovery progress can lag depending on which integrity path the app uses. Make the two integrity paths consistent or document one as violation-only. Prefer adding evaluateRecoveryDay to runSystemIntegrityCheck. P1
REC-EC-13 Recovery protocols exit mid-week without each protocol meeting its own weekly target. Seven clean days require only one counted completion somewhere in the system. A high-frequency recovery protocol can exit still short for the current ISO week and re-enter at week-end. Either set recoveryRestoredAt on protocols leaving .recovery too, or strengthen clean-day rules so active recovery protocols must remain mathematically able to satisfy their own weekly targets. P1
REC-EC-14 Daily Check-In, Plan, logs, and store helpers use older initial-grace logic. UI surfaces may tell the user no work is required when the engine would enforce a shortfall, especially for Tuesday/Wednesday low-frequency session starts. Extract one shared weekly requirement service used by NonNegotiableEngine, CommitmentSystemEngine, CommitmentSystemStore, PlanStore, and DailyCheckInViewModel. P0
REC-EC-15 Multiple suspended protocols exist from legacy data or capacity enforcement. Current pause guard prevents adding another suspended protocol through the popup, but load-time and capacity paths can still leave multiple suspended protocols without matching paused IDs. Add normalization that detects multiple suspended protocols, restores extras if capacity allows, and otherwise records an explicit repair-required state or terminalizes stale paused allocations. P1
REC-EC-16 Retired paused protocol is retired through a call site that does not pass planStore. Domain state is repaired, but paused allocations may remain until full recovery exit or a UI cleanup pass. Make retirement-with-plan-sync the only UI path. Add a defensive PlanStore repair on refresh: paused allocation for retired/missing protocol becomes .skippedDueToRecovery. P1
REC-EC-17 Recovery exit when suspended protocol cannot be restored because capacity is full. resumeSuspendedIfCapacityAllows can leave the protocol suspended. Since no protocol is in recovery, policy may still block that protocol but the system is considered stable. Define product behavior: either force restoration by retiring/asking user to choose, or introduce a non-recovery paused state with clear UI copy. Add simulation for suspended-left-behind after exit. P1
REC-EC-18 Extra completion on a day with no counted completion during recovery. Extra completions do not satisfy clean-day rules, which is probably correct, but this should be explicit because users may interpret "I did work" differently. Add UI copy in recovery check-in: only counted completions move recovery forward. Add simulation confirming extra-only day resets or does not increment streak. P2

4. Recommended Implementation Order

  1. P0 - Recovery catch-up loop: fix evaluateRecoveryDay so skipped days are evaluated one by one.
  2. P0 - Shared weekly requirement logic: remove stale grace/target duplication from store, plan, and check-in surfaces.
  3. P0 - True pause transaction: make domain pause and allocation pause atomic or rollbackable.
  4. P1 - Load-time recovery normalization: repair future evaluation dates, invalid paused IDs, orphaned suspended protocols, and over-limit streaks.
  5. P1 - Exit finalization ownership: make recovery exit cleanup store-owned or event-driven, not only view-owned.
  6. P1 - Candidate policy decision: decide whether recovery protocols can be pause candidates and align code, copy, and tests.
  7. P1 - Re-entry prevention after exit: handle mid-week exit for protocols that were in .recovery, not only suspended protocols.
  8. P2 - Copy and coverage polish: explicit extra-completion rule, UI restart check, strictness checks, and DevOptions recipes.

5. Manual Validation Checklist

  • Restart pending popup: seed In Recovery, force quit before resolving popup, relaunch, confirm popup reappears.
  • Pause sync: choose a pause candidate with future plan allocations; confirm protocol becomes suspended and future allocations become paused.
  • Retire paused protocol: after pausing, retire it after lock end with the UI path; confirm its paused allocations become skipped immediately.
  • Seven clean days: in recovery, complete one counted protocol per day for seven closed days; confirm recovery exits and future paused allocations restore.
  • Dirty skipped days: while in recovery, skip several days at once; confirm the clean-day streak reflects every missed day after REC-EC-01 is implemented.
  • Post-restoration week: exit recovery mid-week with a restored 5x/week session protocol; confirm the first partial week does not immediately re-enter recovery.
