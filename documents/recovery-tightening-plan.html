Fix pause candidate filter — stop offering recovery protocols as pause targets

P0 ⏱ 30 min 📁 CommitmentSystemStore.swift 🔗 Gap G-1 No dependencies

Background

recoveryPauseCandidateIds() currently falls back to returning .recovery protocols when no .active protocols exist. This means a user with two protocols both in recovery could be asked to pause one of the recovering protocols — which provides zero capacity relief and semantically makes no sense. If the fallback is selected, the system ends up with a suspended-but-still-recovering protocol and a dangling paused reference.

What to change

File: CommitmentSystemStore.swift → recoveryPauseCandidateIds(in:)

// BEFORE (lines 862-870):
private func recoveryPauseCandidateIds(in system: CommitmentSystem) -> [UUID] {
    let activeCandidates = sortedNonNegotiableIds(in: system) { $0.state == .active }
    if activeCandidates.isEmpty == false { return activeCandidates }
    let recoveryCandidates = sortedNonNegotiableIds(in: system) { $0.state == .recovery }
    return recoveryCandidates.count > 1 ? recoveryCandidates : []
}

// AFTER — never offer recovery protocols as candidates:
private func recoveryPauseCandidateIds(in system: CommitmentSystem) -> [UUID] {
    return sortedNonNegotiableIds(in: system) { $0.state == .active }
    // If result is empty, caller sets recoveryEntryRequiresPauseSelection = false
}

Also update handleRecoveryTransition() line 845 — the logic for setting recoveryEntryRequiresPauseSelection already uses candidateIds.isEmpty == false, so it will automatically set the flag to false when no active protocols are available. No additional change needed there.

Acceptance checks

  • Create two session protocols. Advance dates to put both in recovery. Open app — popup appears. The selection step shows zero candidates, popup skips directly to confirmed step without offering a selection.
  • Create two protocols where one is active and one is in recovery. Popup shows only the active protocol as a candidate, never the recovering one.

Add rollback to the two-store pause coordinator

P0 ⏱ 45 min 📁 CommitmentSystemStore.swift 🔗 Gap G-2 No dependencies

Background

pauseProtocolForRecoveryWithPlanSync() calls pauseProtocolForRecovery() (which mutates commitment state and persists) then calls planStore.pauseAllocations(). If the plan store call fails or throws after the commitment store has already persisted, the domain shows the protocol as .suspended but all its future allocations remain .active. The plan and domain are now out of sync. The current code has no rollback.

What to change

File: CommitmentSystemStore.swift → pauseProtocolForRecoveryWithPlanSync()

// BEFORE (lines 463-475) — no rollback:
func pauseProtocolForRecoveryWithPlanSync(
    protocolId: UUID,
    planStore: PlanStore,
    referenceDate: Date = Date()
) throws {
    try pauseProtocolForRecovery(protocolId: protocolId)
    guard system.recoveryPausedProtocolId == protocolId,
          nonNegotiable(id: protocolId)?.state == .suspended else { return }
    planStore.pauseAllocations(for: protocolId, referenceDate: referenceDate)
}

// AFTER — snapshot + rollback on plan failure:
func pauseProtocolForRecoveryWithPlanSync(
    protocolId: UUID,
    planStore: PlanStore,
    referenceDate: Date = Date()
) throws {
    // 1. Capture snapshot before any mutation
    let systemSnapshot = system

    // 2. Mutate commitment store (throws on validation failure)
    try pauseProtocolForRecovery(protocolId: protocolId)

    guard system.recoveryPausedProtocolId == protocolId,
          nonNegotiable(id: protocolId)?.state == .suspended else { return }

    // 3. Attempt plan store update; rollback commitment store if it fails
    do {
        try planStore.pauseAllocations(for: protocolId, referenceDate: referenceDate)
    } catch {
        // Restore commitment store to pre-pause state
        system = systemSnapshot
        persistSystem()
        throw error
    }
}
Note: PlanStore.pauseAllocations() currently doesn't throw — it's func pauseAllocations(...). Update its signature to throws so failures can propagate. If it cannot fail in practice, the rollback can remain as a safety net even for a non-throwing call (just remove the try/catch wrapper and call rollback unconditionally on any post-condition mismatch).

Acceptance checks

  • Normal path: pause a protocol — verify both commitment store (protocol = .suspended) and plan store (future allocations = .paused) are updated atomically.
  • After successful pause, verify system.recoveryPausedProtocolId == paused protocol ID.
  • Failure simulation (requires DevOptions or unit test): if pauseAllocations is forced to throw, verify the commitment store rolls back to its snapshot — no protocol should be in .suspended state, recoveryPausedProtocolId should be nil.
  • Verify no partial state is persisted to disk during a failed pause.

Verify plan regulator suspension filter is complete

P0 ⏱ 20 min (audit) 📁 PlanRegulatorEngine.swift, PlanStore.swift 🔗 Gap G-11 No dependencies

Background

Good news: the regulator already handles this partially. Line 92 of PlanRegulatorEngine.swift filters out suspended protocols from the round-robin scheduler: .filter { $0.state != .suspended && remainingSessions(for: $0) > 0 }. Lines 70-81 handle suspended protocols separately — they generate a .warning suggestion with zero confidence, not an actual PlanAllocationDraft. This means the regulator does not draft allocations for suspended protocols.

However, the concern is whether any other code path creates allocations for suspended protocols independently of the regulator — for example, a manual placement flow, a "fill missing weeks" job, or any code that iterates all active protocols without checking the suspension flag.

What to verify (audit checklist)

  • Search all sites in PlanStore.swift that create PlanAllocation objects. For each one, verify the protocol's state is checked and .suspended protocols are excluded.
  • Search for PlanAllocation( constructors across the codebase. Any that don't check state != .suspended are candidates for a bug.
  • Verify that when the regulator's output is applied by PlanStore, only draftAllocations are converted to real allocations — not suggestions of kind .warning.
  • If any allocation-creation path does not check suspension: add guard protocol.state != .suspended else { continue }.

Acceptance checks

  • Pause a protocol for recovery. Advance 3 weeks via DevOptions. Open Plan screen. Verify no new .active allocation rows exist for the suspended protocol in those 3 weeks.
  • Verify the plan regulator's output for those 3 weeks contains only .warning suggestions (not actual allocations) for the suspended protocol.
  • If a manual "place allocation" UI exists, verify it does not allow placement for a suspended protocol (policy gate should block this via CommitmentPolicyEngine.canPlaceAllocation).

Finalize plan allocations atomically on paused-protocol retirement

P1 ⏱ 30 min 📁 CommitmentSystemStore.swift 🔗 Gap G-5 No dependencies

Background

When the paused protocol is retired during recovery, normalizeRecoveryDomain eventually clears recoveryPausedProtocolId. But plan allocations for that protocol stay in .paused state until recovery fully exits (which could be 7+ days later). During this window, the plan screen shows zombie paused slots for a retired protocol with no explanation.

The plan store already has a two-overload finalizeRecoveryAllocationStatuses — the single-protocol overload takes a protocolId and marks all its paused allocations as .skippedDueToRecovery. We just need to call it at the right time.

What to change

File: CommitmentSystemStore.swift → applyPostMutationRecoveryNormalization()

After normalizeRecoveryDomain() runs, inspect its RecoveryNormalizationDecision. If the decision indicates a paused ID was cleared because the protocol is now terminal, immediately finalize that protocol's allocations:

// In applyPostMutationRecoveryNormalization() after normalization runs:
let decision = systemEngine.normalizeRecoveryDomain(in: &updated, referenceDate: referenceDate)

// If a paused protocol became terminal, finalize its allocations immediately
if decision.clearedPausedProtocolId,
   let clearedId = previousPausedId {  // capture before normalization
    planStore?.finalizeRecoveryAllocationStatuses(
        protocolId: clearedId,
        referenceDate: referenceDate
    )
}
You need to capture system.recoveryPausedProtocolId before calling normalizeRecoveryDomain, then use it in the finalization call. The normalization decision's clearedPausedProtocolId: Bool flag tells you whether a clear happened.

Acceptance checks

  • During recovery with a paused protocol: retire that protocol. Immediately open the Plan screen. Verify no allocations for the retired protocol are in .paused state — they should all be .skippedDueToRecovery.
  • Verify recovery itself continues (if other recovering protocols remain above the minimum threshold).
  • Verify finalizeRecoveryAllocationStatuses is not called again for the same protocol when recovery eventually exits.

Guard zero-target week from producing false violations

P1 ⏱ 30 min 📁 NonNegotiableEngine.swift, CommitmentSystemEngine.swift 🔗 Gap G-9, EC-15 No dependencies

Background

isPostRestorationPartialWeek() returns true when a restored protocol has fewer available days than its weekly target, causing effectiveWeeklyTarget() to return 0. This is correct — the target is reduced to zero for that partial week. However, neither evaluateWeekIfNeeded nor evaluateRecoveryDay explicitly guard against the case where effectiveWeeklyTarget == 0. If the end-of-week evaluation compares completionCount (0) < effectiveTarget (0), it evaluates to false so no violation fires — this is accidentally correct. But the clean-day evaluator might separately treat a zero-target day incorrectly.

Making the zero-target guard explicit removes the reliance on accidental correctness and prevents future regressions.

What to change

File 1: NonNegotiableEngine.swift → evaluateWeekIfNeeded() or evaluateWeekCatchUp()

// Add at the top of the weekly violation check, after computing effectiveTarget:
let target = effectiveWeeklyTarget(for: nn, in: weekInterval, calendar: calendar)
guard target > 0 else {
    // Week is fully exempted (creation grace, post-restoration, post-recovery grace).
    // Mark as evaluated so we don't re-check it, but record no violation.
    nn.windows[windowIndex].weeksEvaluated.insert(weekId)
    continue
}

File 2: CommitmentSystemEngine.swift → evaluateRecoveryDay() via requiresCountedCompletionForRecovery()

// In requiresCountedCompletionForRecovery(), when effectiveWeeklyTarget == 0:
let target = effectiveWeeklyTarget(for: nn, in: week, calendar: calendar)
if target == 0 { return false }  // no completion required — week is fully exempted

Acceptance checks

  • Session protocol (3×/week), restored on Saturday (0 feasible days remaining). Advance to Monday (week-end fires). Verify: zero violations recorded for the restoration week, isPostRestorationPartialWeek returned true, week marked evaluated.
  • Session protocol (2×/week), restored on Thursday (3 feasible days). Advance to Monday with 0 completions. Verify: violation IS recorded (3 feasible days ≥ 2 target, so no grace).
  • During recovery, a day where effectiveWeeklyTarget == 0 should count as clean without requiring a completion. Verify streak increments on that day.

One-time migration pass for existing users over new threshold

P1 ⏱ 30 min 📁 CommitmentSystemStore.swift or Locked_INApp.swift 🔗 Gap G-12 No dependencies

Background

Thresholds were lowered from daily: 3→2 and session: 2→1. Recovery entry only fires when updateRecoveryState is called after appending a new violation. Existing users who had 2 daily violations in their window (previously under the threshold of 3) are now over the new threshold of 2, but they won't enter recovery until their next violation is recorded. This is a silent inconsistency — their current state should be recovery but the system shows them as active.

What to change

File: CommitmentSystemStore.swift — add to the store's initialization or to runSystemIntegrityCheck()

private func runThresholdMigrationIfNeeded() {
    let key = "didRunThresholdMigration20260506"
    guard !UserDefaults.standard.bool(forKey: key) else { return }
    defer { UserDefaults.standard.set(true, forKey: key) }

    var updated = system
    for index in updated.nonNegotiables.indices {
        guard updated.nonNegotiables[index].state == .active else { continue }
        var nn = updated.nonNegotiables[index]
        // Let the engine re-evaluate recovery state from current window counts
        nonNegotiableEngine.reevaluateRecoveryStateFromExistingViolations(&nn)
        updated.nonNegotiables[index] = nn
    }
    applySystemUpdate(updated, referenceDate: appClock.now)
}

You will need to add reevaluateRecoveryStateFromExistingViolations() to NonNegotiableEngine. It reads the current window's violation count and calls the existing recovery-state transition logic without appending a new violation:

func reevaluateRecoveryStateFromExistingViolations(_ nn: inout NonNegotiable) {
    guard nn.state == .active else { return }
    guard let windowIndex = currentWindowIndex(for: nn) else { return }
    let count = nn.windows[windowIndex].weeklyViolationCount
    let threshold = recoveryViolationThreshold(for: nn.definition.mode)
    if count >= threshold {
        nn.state = .recovery
    }
}

Call runThresholdMigrationIfNeeded() from runSystemIntegrityCheck() before the main tick, or from the store's init after loading from disk.

Acceptance checks

  • Using DevOptions or a crafted JSON state: create a protocol with 2 daily violations in the current window (state = .active). Launch the app. With the migration sentinel not yet set, the migration runs and the protocol transitions to .recovery on first launch.
  • Launch the app a second time. Migration sentinel is set — migration does not run again. Protocol stays in recovery (already there from last launch).
  • Protocol that has 1 daily violation (below threshold of 2): migration does not transition it.
  • Protocol already in .recovery: migration guard (state == .active) skips it.

Pick best-violation protocol as recovery trigger, not oldest

P2 ⏱ 20 min 📁 CommitmentSystemStore.swift 🔗 EC-8 No dependencies

Background

When two protocols cross the recovery threshold in the same tick, handleRecoveryTransition() sorts newly-recovered protocols by createdAt (oldest first) and picks the first as the trigger. The popup entry step then references that protocol by name. This is misleading — the older protocol may not be the one causing the most damage. The trigger should be the protocol with the highest current-window violation count, as it most clearly explains why recovery was entered.

What to change

File: CommitmentSystemStore.swift → handleRecoveryTransition() lines 833-840

// BEFORE — sorted by creation date (oldest first):
let enteredRecovery = updated.nonNegotiables
    .filter { $0.state == .recovery && previousRecoveryIds.contains($0.id) == false }
    .sorted { lhs, rhs in
        lhs.createdAt != rhs.createdAt
            ? lhs.createdAt < rhs.createdAt
            : lhs.id.uuidString < rhs.id.uuidString
    }

// AFTER — sorted by current window violation count (most violations first):
let enteredRecovery = updated.nonNegotiables
    .filter { $0.state == .recovery && previousRecoveryIds.contains($0.id) == false }
    .sorted { lhs, rhs in
        let lhsCount = lhs.currentWindowViolationCount
        let rhsCount = rhs.currentWindowViolationCount
        if lhsCount != rhsCount { return lhsCount > rhsCount }
        if lhs.createdAt != rhs.createdAt { return lhs.createdAt < rhs.createdAt }
        return lhs.id.uuidString < rhs.id.uuidString
    }

Add a computed property on NonNegotiable (or use an existing helper):

extension NonNegotiable {
    var currentWindowViolationCount: Int {
        windows.last.map { $0.weeklyViolationCount } ?? 0
    }
}

Acceptance checks

  • Create protocol A (3 violations in current window) and protocol B (2 violations). Both transition to recovery in the same tick. Verify recoveryEntryTriggerProtocolId == A.id.
  • Tie scenario: two protocols with equal violations. Verify oldest is chosen (tiebreaker).
  • Single new-recovery protocol: trigger ID still set correctly.

Background — EC-12: Empty system

If the user retires every protocol while the recovery popup is showing, recoveryEntryContext() already guards on system.nonNegotiables.contains(where: { $0.state == .recovery }). If all protocols are retired, no protocol is in .recovery → context returns nil → popup auto-dismisses. This is actually already handled correctly. However, recoveryEntryPendingResolution may still be true in persisted state even though the system is empty, which is a corrupt state.

Add a cleanup call: when the last protocol is retired, call clearRecoveryEntryState() to ensure the flag is reset.

Background — EC-16: Single-protocol stuck in recovery

normalizeRecoveryDomain() checks recoveryEligibleCount < 2 and exits recovery early. recoveryEligibleProtocolIds returns protocols in .active or .recovery state. With exactly 1 protocol in .recovery and nothing else, recoveryEligibleCount == 1 < 2 → recovery exits immediately. This means a single-protocol system should ALREADY exit recovery via normalization on every mutation. Verify this works as expected; if it does, no code change is needed — only a simulation test (Step 12).
