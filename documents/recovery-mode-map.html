2. Current Recovery Mode Flow

Entry Trigger

  1. NonNegotiableEngine.evaluateDailyComplianceIfNeeded appends daily violations and calls updateRecoveryState.
  2. NonNegotiableEngine.evaluateWeekIfNeeded appends missed-weekly violations and calls updateRecoveryState.
  3. updateRecoveryState flips protocol state to .recovery when the current window hits threshold: daily = 3, session = 2.

System Transition

  1. Store mutations eventually flow through CommitmentSystemStore.applySystemUpdate.
  2. handleRecoveryTransition detects first entry into recovery, then sets:
    recoveryEntryPendingResolution = true, recoveryEntryRequiresPauseSelection, recoveryEntryTriggerProtocolId, clears recoveryPausedProtocolId, resets recoveryCleanDayStreak, backdates lastRecoveryEvaluationDay.

Presentation

  1. MainAppView.evaluateRecoveryEntryPresentation checks store.recoveryEntryContext().
  2. If context exists, it sets AppRouter.presentRecoveryEntry.
  3. RecoveryModePopup is rendered above the app shell.

Detailed Enter - Resolve - Exit Sequence

  1. Detection: recovery is entered at the protocol level inside NonNegotiableEngine.updateRecoveryState after violations have been counted into the current 14-day window.
  2. Pending entry state: on the next store update, CommitmentSystemStore.handleRecoveryTransition marks entry as unresolved and determines whether the user must pause another protocol.
  3. Popup preparation: CommitmentSystemStore.recoveryEntryContext returns the trigger protocol, paused protocol, whether pause selection is required, and candidate protocol IDs drawn from .active and .recovery states.
  4. User flow: RecoveryModeViewModel.refresh ranks options using current-window violations, weekly load, and active planned load. The popup advances through .entry - .selectProtocol - .confirmed.
  5. Pause action: if selection is required, confirmPauseSelection calls CommitmentSystemStore.pauseProtocolForRecovery and then PlanStore.pauseAllocations so future active allocations for that protocol become .paused.
  6. Recovery confirmation: completeFlow calls completeRecoveryEntryResolution, which clears the pending-entry flags and trigger reference.
  7. During recovery:
    policy planning cockpit daily check-in
    • CommitmentPolicyEngine.canCreate denies new protocol creation while any protocol is in .recovery.
    • CommitmentPolicyEngine.canRecordCompletion and canPlaceAllocation deny actions on suspended protocols.
    • PlanStore.shouldApplyRecoveryStrictness reduces placement freedom when any protocol is recovering.
    • PlanRegulatorEngine sorts recovery protocols first and applies a strong same-day isolation penalty.
  8. Clean-day evaluation: CommitmentSystemEngine.evaluateRecoveryDay evaluates the most recently closed day. A clean day requires no violation and either:
    • at least one counted completion somewhere among active/recovery protocols, or
    • no counted completion was actually required under grace/weekly-feasibility logic.
  9. Normal exit path: after 7 clean days, evaluateRecoveryDay turns all .recovery protocols back to .active, resumes suspended protocols if capacity allows, and resets the streak.
  10. Normalization exit path: normalizeRecoveryDomain also exits recovery early if fewer than 2 recovery-eligible protocols remain after retirement or state repair.
  11. Plan cleanup on exit: MainAppView calls PlanStore.finalizeRecoveryAllocationStatuses with the active protocol IDs that should be restored. Future paused allocations become .active; past or non-restorable ones become .skippedDueToRecovery.

5. Missing Or Suspicious Areas

Area Why It Looks Risky Files
User can potentially pause a recovery protocol, not just an active one recoveryEntryContext returns candidates from both .active and .recovery, and pauseProtocolForRecovery explicitly allows pausing either state. That conflicts with popup copy that says “Select exactly one active protocol,” and it may allow the only recovering protocol to be suspended without a normalization pass. CommitmentSystemStore.swift (403-418, 421-439), RecoveryModePopup.swift (176-177)
Pause action is split across two stores The real recovery-entry pause is only complete if both CommitmentSystemStore.pauseProtocolForRecovery and PlanStore.pauseAllocations are called. That coupling lives in RecoveryModeViewModel.confirmPauseSelection, not in a single transactional API, so a future caller could update one side but forget the other. RecoveryModeViewModel.swift (137-143), CommitmentSystemStore.swift (421-439), PlanStore.swift (329-346)
No formal XCTest or UI-test coverage All current coverage is simulation-style printing and verification helpers. There is no test target in the project file and no dedicated UI coverage for popup presentation, selection flow, or exit cleanup wiring. 6. Bottom Line

What is clearly implemented today: recovery entry via rolling violations, pending recovery-entry resolution, optional protocol pausing, 7 clean-day recovery exit, retirement-driven recovery normalization, paused allocation cleanup, recovery-aware policy blocking, recovery-aware regulator behavior, and recovery-aware cockpit/daily-check-in/log surfaces.

What is weakest today: formal automated testing, popup/router flow coverage, and consistency around which protocol states are valid pause candidates during recovery entry.
